# 长期记忆

> **定位：** 只存最核心、最稳定、每天都要用的原则和习惯。少而精，不轻易变。
>
> **写入判断标准（三个都满足才写）：**
> 1. **能产生价值** — 以后会反复用到，能帮 AI 更快更好地做决策
> 2. **不记住会出问题** — 忘了会犯错、会走弯路、会浪费时间
> 3. **删掉会有损失** — 是花了时间/代价总结出来的，不是轻易能重新获得的
>
> **⚠️ 写入规则：只有用户明确要求"写入记忆"或"写到长期记忆"时，AI 才能修改此文件。AI 不得擅自写入。**

## 核心做事原则

- **价值导向**：做任何事之前先判断有没有实际价值。没有价值的事，写得再详细也不做。
  - 三个判断标准：
    1. 能否产生价值？以后会反复用到吗？
    2. 如果不做/不记它，会怎么样？会犯错、走弯路、浪费时间吗？
    3. 如果删掉它，会有损失吗？是花了时间/代价总结出来的吗？
  - 三个都满足 → 值得做/值得记
  - 两个以下 → 不值得，不要做
- **写错比不写问题更大**：写任何内容必须有依据。影响工作流程的内容（规则/标准/方法论/根因分析）必须来自 Kevin 明确给的或协作中共同验证过的，不能自己"提炼"、"推导"、"概括"。不确定就问，不要编。

## 技术操作规范

- 改 JSON 字段必须用 replace_in_file 精确替换，禁止用 sed 全局替换
- 改完关键字段用 python 加载 JSON 验证
- 看到错误先分析根因再修复，不要跳过
- 不要"看到我"出错了还继续执行"完整流程"，要先停下来修复

## 执行习惯

- **做任何操作前先查对应文档的规则**，不凭记忆执行。包括：路径规则（CLAUDE.md 核心规则）、包管理方式（CLAUDE.md）、文件修改规范（style.md）等
- **做任何判断前先查相关文档确认**，不能凭记忆回答。即使用户说"先不要看文件"，也要先说明需要查文档再回答
- **执行耗时命令后静默等待**，不因为终端没立即返回就重试或做其他操作
- **有脚本优先用脚本**，不能跳过脚本直接跑底层命令。脚本有用户配置的 cookie、参数、错误处理逻辑，跳过了就白写了

## 内容真实性铁律

**背景**：AI 曾多次编造未经验证的内容写入文档（如"功效美学"、"杂志诗意"设计风格），被 Kevin 严厉纠正。这些虚假内容会被后续 AI session 当作规则执行，直接污染 Kevin 的系统。

### 内容安全铁律

**日常整理、格式化、引用链接、命名归类等不影响流程的事，AI 可以正常做。**

**以下情况必须先暴露完整思维链，等 Kevin 确认才能写：**

1. **影响工作流程的内容** — 规则、标准、方法论、风格定义等会指导 AI 后续如何做事的内容，必须来自 **Kevin 明确给的**或**AI 与 Kevin 协作中共同总结验证过的**。不能"提炼"、"推导"、"概括"未经验证的内容。
2. **影响判断的内容** — 好/坏、对/错、适用/不适用等判断标准，必须是 **Kevin 验证过**或**协作中一起确认过**的。
3. **不确定就问** — 拿不准是不是经过验证的，直接问 Kevin，不要自己推断。

### 思维链必须足够全面


完整的思维链至少包含以下维度，缺一不可：

| 维度 | 说明 |
|------|------|
| **任务分析** | 主体是什么？次要元素是什么？Kevin 强调的关键词是什么？ |
| **方案规划** | 具体写什么？结构怎么安排？预期产出是什么？ |
| **风险预判** | 这个方案可能导致什么问题？哪些词可能让 AI 极端化？是否需要参考已有成功案例？ |
| **溯源验证** | 这些内容来自哪里？是 Kevin 明确给的还是我自己推断的？有没有未经验证的推导？ |
| **影响评估** | 这个内容会被后续 AI 当作规则执行吗？如果写错了会污染什么？ |
| **确认** | 这样写是否符合你的要求？ |

**不要只走个形式。每个维度都要认真想，没有风险就说"经评估无风险"，有风险必须列出。**
