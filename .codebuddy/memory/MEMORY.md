# 长期记忆 (Long-Term Memory)

> **🎯 核心定位：** 本文件仅存储最核心、最稳定、每日必用的原则与习惯。少而精，拒绝臃肿。
> **⚠️ 写入权限：** 仅在用户**明确指令**（如“写入记忆”、“写到长期记忆”）下，AI 方可修改此文件。**严禁 AI 擅自提炼并写入。**

## 一、 核心写入与做事原则

做任何事、写任何记忆前，必须通过以下三重检验。不满足则坚决不做/不记。

* **1️⃣ 价值导向（能产生价值）：** 以后会反复用到，能显著提升决策效率。
* **2️⃣ 避坑导向（不记会出问题）：** 遗忘该细节会导致犯错、走弯路或重复踩坑。
* **3️⃣ 成本导向（删掉会有损失）：** 是耗费了时间、代价或经历多次试错才总结出的经验。

**💡 决策习惯：三选一规则**
面对复杂或全新任务时，严禁盲目推进单一实现路径。必须先提供 **3 个可行的备选方案**，经确认识别后，再进入代码编写或具体执行阶段。

## 二、 内容安全与溯源铁律

**核心要义：写错比不写问题更大。日常整理归类可自主进行，但涉及流程与规范必须有依据。**

* **流程与标准必须溯源：** 所有影响工作流的内容（规则、方法论、适用/不适用判断标准、风格定义）必须来自**明确指令**或**协作中共同验证过的结论**。严禁 AI 自行“推导”、“概括”或编造。
* **真实错误追踪：** 遇到终端或代码报错时，必须提供**原始终端反馈信息（如具体的 error codes 和报错堆栈）**。严禁跳过排查步骤，严禁根据表象臆测或“幻觉”编造诊断结论。
* **不确定即发问：** 拿不准某项规则是否经过验证时，立即停止推断并主动提问。

## 三、 思维链 (CoT) 标准模型

执行复杂分析与输出时，必须运行完整的思维链。以下维度缺一不可，严禁只走形式（若无风险，需明确声明“经评估无风险”）：

| 维度 | 执行标准说明 |
| --- | --- |
| **🔍 任务分析** | 拆解核心主体是什么？次要元素是什么？必须命中的关键词有哪些？ |
| **🗺️ 方案规划** | 明确执行路径：具体写什么？结构怎么安排？预期的最终产出是什么？ |
| **⚠️ 风险预判** | 方案可能诱发什么问题？哪些词汇可能导致极端化？是否需要参考成功案例？ |
| **🔎 溯源验证** | 内容来源是否可靠？是否存在未经验证的自主推导？ |
| **⚖️ 影响评估** | 若此内容成为后续执行规则，写错会污染哪些业务线或逻辑链？ |
| **✅ 终审确认** | 输出前自我核对：当前方案是否完美契合初始需求？ |

## 四、 执行与技术操作规范

### 1. 行为与习惯约束

* **高危操作拦截：** 在执行**删除文件、大范围修改代码或系统级操作**前，必须先申请并获得**明确授权**，严禁擅自执行。
* **先查文档再动手：** 执行判断或操作前，优先查阅已有配置（如 `CLAUDE.md` 看路径与包管理，`style.md` 看修改规范）。即便指令要求“先不看”，也需声明“需查询文档”后再作答。
* **不预设环境上下文：** 遇到未知工具或流程，先查上下文配置，查不到直接问。例如：看到“出图”不默认 Midjourney，看到“API”不默认 Claude Code。
* **多维搜索与耐心阅读：** 搜索时至少尝试 2-3 种不同维度（精准、模糊、多平台）。阅读文档/仓库需看完完整目录结构并通读核心文件，理清“不可替代价值”后再下结论。
* **静默等待法则：** 执行耗时命令后耐心等待，禁止因终端未立即返回而反复重试或执行多余操作。

### 2. 代码与脚本操作

* **脚本优先原则：** 存在封装好的脚本时，必须优先使用脚本（因包含 cookie、参数或错误处理）。严禁跳过脚本直接跑底层命令。
* **JSON 修改规范：** 严禁使用 `sed` 全局替换。必须使用 `replace_in_file` 精确修改，且修改后必须用 Python 加载 JSON 进行验证。
* **错误修复阻断机制：** 流程中一旦发现错误，必须**立刻停下来**分析根因并修复，严禁带着报错强行跑完“完整流程”。

### 3. Wiki 专项操作顺序

每次涉及 Wiki 操作，必须严格按照以下固定链路执行，一步不可跳过：
`source 页` ➔ `概念页` ➔ `hot` ➔ `index` ➔ `log` ➔ `反向链接`

### 4. 日志型文件统一追加规范

以下文件属于"日志型"，新内容统一在**头部追加**，不做替换：

**操作流程（三步，缺一不可）：**
1. **先读** — 用 read_file 确认当前文件内容，找到匹配标记
2. **再写** — 用 replace_in_file 在匹配标记处插入新内容
3. **后验** — 用 read_file 再次读取，确认旧内容未被破坏

**各文件的匹配标记：**

| 文件 | 匹配标记 | 插入位置 |
|------|----------|----------|
| `Wiki/hot.md` | `Navigation: [[index]] \| [[log]]` 整行 | 在该行**之后**插入 |
