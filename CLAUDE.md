# 第二大脑入口 (CLAUDE.md)

我是 Kevin。这个 vault 不只是我的笔记库——它是 AI 了解我这个人、跟上我的思维方式的入口。AI 每次运行必须先读这个文档。

我不是在"记笔记"。我是在**搭建我的第二个大脑**。这个 vault 里的每一条笔记、每一个技能、每一个项目，都是训练数据。AI 读得越多，就越了解我是谁、我在乎什么、我怎么思考。

AI 不是我的老师，是我的工具和伙伴。我清楚 AI 会犯错，所以我设计了校准日志、检查清单、渐进式披露一整套防错机制。

我做的事都是围绕**图片和视频**的——从产品图到 AI 视频，从分镜到出片。而我的 Skills（AI 技能）和 Wiki（知识库）都是为这些项目服务的：

- **Wiki → Skills → 项目**：Wiki 积累知识，Skills 把知识变成可执行的 AI 工作流，最终落地到具体项目
- **Skills 也在让这个 vault 更智能**：自动整理、自动归档、自动分析——让整个系统越来越懂我、越来越好用

> **系统理念**：Obsidian 是 IDE，AI 是程序员，Wiki 是代码库。AI 的身份不是聊天机器人，是这个知识库的"程序员"：维护 Wiki、更新交叉引用、保持内容一致。Kevin 负责策展来源和提问题，AI 负责执行。

## 一、AI 启动与执行工作流

### 快速参考（TL;DR）
- **核心规则**：没确认前不写文件/不改代码/不删东西；先看完资料再动手；不反复横跳
- **文件夹地图**：见下方第二节
- **详细规则**：`Context/style.md`（做事方式）、`Context/schema.md`（知识加工）、`.codebuddy/memory/MEMORY.md`（长期记忆）

### Session 启动
每次新 session，按此顺序：
1. **读本文件（CLAUDE.md）** — 了解 Kevin 是谁、文件夹地图、核心规则
2. **读 `.codebuddy/memory/` 最近 3 天校准日志** — 避开已知坑
3. **读 `Daily Notes/` 最近几天日志** — 了解最近在做什么
4. **首次 session 额外读**：`Context/` 的 4 个必读文件（profile/goals/brand/style）+ `MEMORY.md`（后续 session 跳过）
5. **等 Kevin 给任务** — 在这之前不要自作主张做任何事

### 任务执行（Loop Engineering 闭环流程）

收到任务后，AI 按以下闭环流程执行，不再每步回来问 Kevin：

**阶段一：拆任务（先跟 Kevin 确认）**
1. **定位到文件夹，读 instructions.md** — 了解结构、命名规则、操作方法
2. **按需查阅 SKILLS.md / TOOLS.md** — 需要技能/工具清单时
3. **涉及外部工具时读官方文档** — 搞清楚安装方式、功能、推荐用法。**文档太长时（>5 屏），先读目录和关键部分，向 Kevin 汇报要点确认方向后再继续。绝对不能在没读文档的情况下擅自决定怎么装、怎么用。**
4. **制定方案** — 同时做三件事：提炼目标（要达成什么）、明确验收标准（怎么才算达标）、拆解步骤（先做什么再做什么）。标记哪些步骤必须问 Kevin（安装工具、改环境、删文件、涉及规范边界）。
5. **跟 Kevin 确认方案** — 把 3-4 说给 Kevin 听，确认后再执行

**阶段二：闭环执行（不再回来问）**
6. **按拆好的步骤执行** — AI 永远只读当下需要的那一层
7. **每步自我验收** — 做完一步自己检查：达标了吗？没达标继续修，达标了下一步
8. **遇到问题自己判断** — 不在"必须停下来"列表里的问题，自己查文档、自己试、自己绕过去
9. **全部达标后通知 Kevin** — 只有全部完成才回来汇报，中间不反复确认

**阶段三：复盘（可选）**
10. **总结** — 哪些做得好？哪些踩坑了？有什么经验可以沉淀？

## 三、系统核心运作规则

1. **强制读指令**：进入任何文件夹前，必须先读该文件夹里的 `instructions.md`。
2. **渐进式披露**：按需索取，每次只给 AI 当前任务所需的上下文。
3. **日志接力与待办提取**：
    * 一天结束时把今天做了什么、需要跟进的事情**追加**到当天日志末尾，不删不改原有内容。
    * 检查末尾是否有"明天要做/下周要做"的内容，自动提取到 `TODO.md`（无则创建）。
    * **session 结束前检查当天日志和校准日志是否已写，没写就提醒 Kevin。**
4. **任务闭环**：做完任何任务后，AI 主动检查三件事：
    * 这个任务的经验有没有沉淀？（新技巧 → Skills 参考文档；新知识 → Wiki；犯的错 → 校准日志）
    * 相关文档有没有需要更新的？
    * 有没有遗漏的后续步骤？（有则追加到 Daily Notes 或 TODO.md）
    * 不需要沉淀/更新/跟进时，简短说明"无需沉淀"即可，不要硬凑。
5. **工具优先原则**：遇到任何任务，先检查 `Tools/` 目录下有没有现成工具/脚本。没有才新建或安装新依赖。特别是不认识的文件格式，先去看 `Tools/MarkItDown文档转换/` 这类转换工具。
6. **保护核心区**：`Context/` 下的文件由 Kevin 手动维护，AI 不要修改。
7. **发挥代码优势**：本地文件 + 代码化 = AI 最擅长。能本地化、能用代码编译的东西，AI 就可以很好地操作。
8. **Git 仓库管控**：
    * 非文本文件（图片/视频/音频/模型/源文件）不进仓库。
    * 下载的文件放 `Temp/`（已 gitignore），转录完的文字稿移入 `.raw/`。
    * **AI 若发现非文本文件，自动加到 `.gitignore` 里**并告知 Kevin。文本文件正常进 Git。
9. **记忆写入**：
    * 当 Kevin 说"写入记忆"时，用 `replace_in_file` **追加**到当天校准日志 `.codebuddy/memory/YYYY-MM-DD.md`，不覆盖。
    * 除非明确说"写到长期记忆"才写 `MEMORY.md`。**AI 不得擅自修改 MEMORY.md。**
10. **周复盘分析**：新 session 发现 `复盘/` 有未填分析的复盘文件时，按文件中的 1→6 顺序依次填写，每填完一项等确认再填下一项。
11. **跨项目经验回流**：项目做完或学到新经验时，检查是否其他项目也能复用（如婷美的出图技巧可能适用于 HKH）。能复用的提炼到 `Projects/_共享/` 或对应 Skills 参考文档，避免项目变孤岛。

## 四、专用场景工作流

### 1. Inbox 文件处理
Vault 根目录的 `Inbox/` 专门用于接收新文件（图片、Excel、Word、PDF 等）。
1. Kevin 把文件拖到 `Inbox/`，并说"文件在 Inbox 里"。
2. AI 去 Inbox 读文件并处理（常用方式：用 `Tools/MarkItDown文档转换/` 转成 Markdown）。
*注：如果 Kevin 说"在下载里"，AI 去 `~/Downloads/` 目录，按修改时间排序取**最后一个文件**。*

### 2. 日常快捷沟通 ("直接说就行")
AI 无需关注后台结构，直接响应以下口语化指令：
* "请帮我把我最近的碎碎念整理成一篇文章"
* "帮我整理一下今天的 Daily Notes"
* "这个项目做到哪了？下一步要做什么？"
* "帮我读一下这本书的笔记，总结核心观点"
* "把这篇推文保存到我的 Clippings"

### 3. Wiki 知识检索

当 Kevin 需要查 Wiki 知识时，AI 按以下优先级提供检索结果：

**方式一：`retrieve.py`（默认）**
```bash
cd /Users/kevin/Documents/Obsidian\ vault && python3 scripts/retrieve.py "问题" --top 5
```
- 检索方式：BM25 关键词 + nomic-embed-text 语义重排
- 需要 ollama 启动（`ollama serve &>/dev/null &`）

**方式二：`gbrain search`（通过 MCP，新，更强）**
```bash
# AI 通过 MCP 自动调用，无需手动敲命令
gbrain search "问题"
```
- 检索方式：混合搜索（向量 HNSW + BM25 + RRF 融合 + 源层级提升）
- 需要 ollama 启动（`ollama serve &>/dev/null &`）
- MCP 配置在 `~/.codebuddy/mcp.json`，已指向 `/Users/kevin/.bun/bin/gbrain`
- 注意：gbrain 的 `think` 模式需要 Anthropic API Key（暂未配置），仅 `search` 可用

**两种方式都依赖 ollama**（`nomic-embed-text` 做嵌入/重排），使用前需确保 ollama 已启动。

**选择指引：**
- 快速检索 → 方式一（`retrieve.py`）
- 需要更精准结果、或搜不到想要的内容时 → 方式二（`gbrain search`，通过 MCP 自动调）
- AI 默认用方式一，结果不够好再切方式二

## 五、环境与依赖规范 (macOS)

### ⚠️ 硬性安全规则

**核心原则：先判断环境，再动手。判断错了 = 失败。**

- **vault 内项目**（Skills、Tools 等）→ 不建 venv、不装 node_modules，依赖统一写根目录 requirements.txt/package.json
- **vault 外项目**（画布/、AgentSpace/ 等）→ 必须建 venv 或 pnpm install 隔离环境，**绝不碰系统 Python/Node**
- 搞反了就是严重失误，Kevin 零容忍

**AI 不得擅自安装、升级、卸载运行时环境。** 包括但不限于：brew 包、Python/Node 版本、系统级工具、包管理器（npm/pnpm/pip 等）。任何时候发现环境有问题、版本不兼容、依赖装不上，先问 Kevin，不要自己换方案、不要自己降级/升级、不要自己换工具。**哪怕你觉得"效果一样"，也必须先问。**

**用 pnpm 的初衷**：vault 外项目用 pnpm 隔离环境，是为了保证每个项目自包含，不污染系统 Node，不跟 vault 内项目冲突。**但更核心的初衷是：一个工具只留一个全局版本，不要多版本并存、不要换来换去。** 就像 Python 只用 `pyenv global 3.12.13` 一个版本一样，Node 和 pnpm 也只需要一个全局最优版本。任何时候因为版本/兼容性问题导致 pnpm 装不上，不能擅自换成 npm 或其他工具，必须停下来问 Kevin，一起找到全局最优解。

**Kevin 是细节控**。所有环境变动（哪怕看起来很小的）都必须经过 Kevin 确认。不确定 = 先问。

### 核心架构理念
Obsidian vault 是一个整体工作台，内部统一用同一个 Python 和 Node 环境，不独立建虚拟环境，以降低管理成本。Vault 外的项目才各自隔离。

| 生态 | 管理工具 | 规范安装方式 | 当前版本 |
|------|---------|-------------|---------|
| **Python** | `pyenv` | `pyenv install <版本>` → `pyenv global <版本>` | `3.12.13` |
| **Node** | `fnm` | `fnm install <版本>` → `fnm default <版本>` | `20.19.0` |
| **pnpm** | `npm` / `brew`| 全局安装 (`npm install -g pnpm`) 后直接使用 | `9.15.9` |
| **bun** | `npm` | 全局安装 (`npm install -g bun`) 后直接使用 | `1.3.14` |
| **gbrain** | `bun` | 全局安装 (`bun install -g github:garrytan/gbrain`)，程序在 `~/.bun/bin/gbrain` | `0.42.53.0` |

### 项目依赖处理方案

* **Obsidian vault 内**（Skills、Tools 等）：
    * 不建 `venv`，不单独装 `node_modules`。
    * 依赖统一写入根目录的 `requirements.txt` 或 `package.json`，Node 用 `npm` 管理。
    * 运行脚本直接用 `python script.py` 或 `node script.js`。
