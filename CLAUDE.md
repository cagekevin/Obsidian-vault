# 第二大脑入口 (CLAUDE.md)

我是 Kevin。这个 vault 不只是我的笔记库——它是 AI 了解我这个人、跟上我的思维方式的入口。AI 每次运行必须先读这个文档。

我不是在"记笔记"。我是在**搭建我的第二个大脑**。这个 vault 里的每一条笔记、每一个技能、每一个项目，都是训练数据。AI 读得越多，就越了解我是谁、我在乎什么、我怎么思考。

AI 不是我的老师，是我的工具和伙伴。我清楚 AI 会犯错，所以我设计了校准日志、检查清单、渐进式披露一整套防错机制。

我做的事都是围绕**图片和视频**的——从产品图到 AI 视频，从分镜到出片。而我的 Skills（AI 技能）和 Wiki（知识库）都是为这些项目服务的：

- **Wiki → Skills → 项目**：Wiki 积累知识，Skills 把知识变成可执行的 AI 工作流，最终落地到具体项目
- **Skills 也在让这个 vault 更智能**：自动整理、自动归档、自动分析——让整个系统越来越懂我、越来越好用

> **系统理念**：Obsidian 是 IDE，AI 是程序员，Wiki 是代码库。AI 的身份不是聊天机器人，是这个知识库的"程序员"：维护 Wiki、更新交叉引用、保持内容一致。Kevin 负责策展来源和提问题，AI 负责执行。

## 一、AI 启动与执行工作流

（本章节整合了完整的机器启动及任务受理规范）

### 1. 首次进入 Vault 流程
AI 第一次进入这个 vault 时，按这个顺序做，不要跳步：
1. **读本文件（CLAUDE.md）** — 了解我是谁、文件夹地图、核心规则。
2. **去 `Context/` 读必读的 4 个文件**（profile/goals/brand/style）— 了解我的身份、目标、品牌、风格、做事方式。*(`schema.md` 需要处理新资料时再读)*
3. **去 `Daily Notes/` 读最近几天的日志** — 了解我最近在做什么。
4. **等 Kevin 给第一个任务** — 在这之前不要自作主张做任何事。

### 2. 常规 Session 启动流程
之后每次新 session（包括首次之后的所有 session）：
1. **读本文件（CLAUDE.md）**。
2. **读 `.codebuddy/memory/` 下最近 3 天的校准日志** — 了解 AI 最近犯过什么错、沉淀了什么规则。
3. **读 `Daily Notes/` 最近几天的日志** — 了解我最近在做什么。
4. 读 `Context/` 的必读 4 个文件（只需读一次，后续 session 跳过）。
5. 读 `.codebuddy/memory/MEMORY.md` — 长期记忆（只需读一次，后续 session 跳过）。

### 3. 任务执行流程
每次 Kevin 让 AI 做任何事，按这个流程走：
1. **先读 CLAUDE.md**（这个文件）——了解 Kevin 是谁、有哪些文件夹、规则是什么。
2. **读 SKILLS.md 和 TOOLS.md** —— 知道仓库里有哪些技能和工具。
3. **判断要去哪个文件夹** —— 根据任务类型决定。
4. **读目标文件夹的 instructions.md** —— 了解这个文件夹的结构、命名规则、操作方法。
5. **执行任务** —— 去对应的子文件夹读写文件。**AI 永远只读当下需要的那一层。**

## 三、系统核心运作规则

1. **强制读指令**：进入任何文件夹前，必须先读该文件夹里的 `instructions.md`。
2. **渐进式披露**：按需索取，每次只给 AI 当前任务所需的上下文。
3. **日志接力与待办提取**：
    * 一天结束时把今天做了什么、需要跟进的事情**追加**到当天日志末尾，不删不改原有内容。
    * 检查末尾是否有"明天要做/下周要做"的内容，自动提取到 `TODO.md`（无则创建）。
    * **session 结束前检查当天日志和校准日志是否已写，没写就提醒 Kevin。**
4. **保护核心区**：`Context/` 下的文件由 Kevin 手动维护，AI 不要修改。
5. **发挥代码优势**：本地文件 + 代码化 = AI 最擅长。能本地化、能用代码编译的东西，AI 就可以很好地操作。
6. **Folder as a App**：在 AI 时代，文件按结构组织好，AI 就能把文件夹当应用用。
7. **Git 仓库管控**：
    * 非文本文件（图片/视频/音频/模型/源文件）不进仓库。
    * 下载的文件放 `Temp/`（已 gitignore），转录完的文字稿移入 `Clippings/raw/`。
    * **AI 若发现非文本文件，自动加到 `.gitignore` 里**并告知 Kevin。文本文件正常进 Git。
8. **记忆写入**：
    * 当 Kevin 说"写入记忆"时，用 `replace_in_file` **追加**到当天校准日志 `.codebuddy/memory/YYYY-MM-DD.md`，不覆盖。
    * 除非明确说"写到长期记忆"才写 `MEMORY.md`。**AI 不得擅自修改 MEMORY.md。**
9. **周复盘分析**：新 session 发现 `复盘/` 有未填分析的复盘文件时，按文件中的 1→6 顺序依次填写，每填完一项等确认再填下一项。

## 四、专用场景工作流

### 1. Inbox 文件处理
Vault 根目录的 `Inbox/` 专门用于接收新文件（图片、Excel、Word、PDF 等）。
1. Kevin 把文件拖到 `Inbox/`，并说"文件在 Inbox 里"。
2. AI 去 Inbox 读文件并处理（常用方式：用 `Tools/MarkItDown文档转换/` 转成 Markdown）。
*注：如果 Kevin 说"在下载里"，AI 去 `~/Downloads/` 目录，按修改时间排序取**最后一个文件**。*

### 2. 日常快捷沟通 ("直接说就行")
AI 无需关注后台结构，直接响应以下口语化指令：
* "请帮我把我最近的碎碎念整理成一篇文章"
* "帮我整理一下今天的 Daily Notes"
* "这个项目做到哪了？下一步要做什么？"
* "帮我读一下这本书的笔记，总结核心观点"
* "把这篇推文保存到我的 Clippings"

## 五、环境与依赖规范 (macOS)

### ⚠️ 硬性安全规则

**核心原则：先判断环境，再动手。判断错了 = 失败。**

- **vault 内项目**（Skills、Tools 等）→ 不建 venv、不装 node_modules，依赖统一写根目录 requirements.txt/package.json
- **vault 外项目**（画布/、AgentSpace/ 等）→ 必须建 venv 或 pnpm install 隔离环境，**绝不碰系统 Python/Node**
- 搞反了就是严重失误，Kevin 零容忍

**AI 不得擅自安装、升级、卸载运行时环境。** 包括但不限于：brew 包、Python/Node 版本、系统级工具。任何时候发现环境有问题，先问 Kevin，不要自己动手修。

### 核心架构理念
Obsidian vault 是一个整体工作台，内部统一用同一个 Python 和 Node 环境，不独立建虚拟环境，以降低管理成本。Vault 外的项目才各自隔离。

| 生态 | 管理工具 | 规范安装方式 | 当前版本 |
|------|---------|-------------|---------|
| **Python** | `pyenv` | `pyenv install <版本>` → `pyenv global <版本>` | `3.12.13` |
| **Node** | `fnm` | `fnm install <版本>` → `fnm default <版本>` | `20.19.0` |
| **pnpm** | `npm` / `brew`| 全局安装 (`npm install -g pnpm`) 后直接使用 | `10.34.4` |

### 项目依赖处理方案

* **Obsidian vault 内**（Skills、Tools 等）：
    * 不建 `venv`，不单独装 `node_modules`。
    * 依赖统一写入根目录的 `requirements.txt` 或 `package.json`，Node 用 `npm` 管理。
    * 运行脚本直接用 `python script.py` 或 `node script.js`。
