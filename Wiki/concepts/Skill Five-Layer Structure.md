# Skill Five-Layer Structure

Skill 的五层架构。一个完整的、好用的 Skill，不是一坨 prompt，而是五层结构的叠加。

---

## 五层结构

### 第一层：方法论层
**是什么、为什么。** 这个 Skill 背后的理论、原则、框架。

- 核心思想是什么？
- 为什么要这么做？
- 有哪些底层原则？

**例子：**
- 图片设计 Skill → 描述比重原则、克制原则、参考图优先原则
- 剧本写作 Skill → Show don't tell、三幕式、英雄之旅
- 出图工作流 → 系统化思维、解耦思维、防错优先

**这一层放在 Wiki 里。** 通用方法论，跨项目复用。

---

### 第二层：写作/执行规则层
**怎么做。** 具体的规则、注意事项、禁忌、最佳实践。

- 有哪些必须遵守的规则？
- 哪些地方容易出错？
- 有哪些最佳实践？

**例子：**
- 图片设计 → 不要用程度词、次要元素描述要少、能垫图就垫图
- 剧本写作 → 每句台词都要推动剧情、每个场景都要有价值变化
- 出图 → 提示词分层拼装、自审三件套（逻辑矛盾→极端化→主次比例）

**这一层放在 Skill 内部。** 每个 Skill 有自己的规则。

---

### 第三层：工作流层
**怎么和用户交互。** 用户的不同意图对应什么处理流程。

- 用户可能会要什么？（有哪些意图）
- 每个意图对应什么处理步骤？
- 信息不够时怎么问？
- 用户否定时怎么应对？
- 出问题时怎么兜底？

**核心：意图识别 + 对应流程。**

**例子：**
- 剧本写作 Skill → 7 个意图（要场景、给修改、提想法、否定、要剪长度、要审计、信息不足）
- 出图工作流 → 4 步必选 + 可选插拔（场景设计、物料包、返工、沉淀）
- 电商主图 Skill → 双模引擎（白底图 + 场景图）

**这一层是 Skill 的灵魂。** 决定了用户用起来顺不顺手。

---

### 第四层：输出工具层
**交付什么格式。** 最终产出的格式、模板、导出方式。

- 输出什么格式？（Markdown / JSON / 图片 / 视频）
- 有哪些模板？
- 怎么导出？
- 输出到哪里？

**例子：**
- 剧本写作 → 角色圣经模板、世界观模板、分镜模板
- 电商详情页 → 逐屏文案模板、图片规格说明
- 出图 → 提示词格式、输出目录结构

**这一层决定了交付物的质量和可用性。**

---

### 第五层：质量控制层
**节奏、删减、验收、自检。** 怎么保证输出质量，怎么控制节奏，怎么验收。

- 怎么自检？
- 验收标准是什么？
- 节奏怎么控制？
- 太长了怎么删？
- 出问题了怎么修？

**例子：**
- 剧本写作 → 节拍计算、删减原则、审计清单
- 出图 → 自审三件套（逻辑矛盾→极端化→主次比例）
- 代码 Skill → 测试用例、lint 检查

**这一层决定了 Skill 的可靠性。**

---

## 各层的关系

```
方法论层（为什么）
    ↓
执行规则层（怎么做）
    ↓
工作流层（怎么交互）
    ↓
输出工具层（交付什么）
    ↓
质量控制层（怎么保证质量）
```

- **上层指导下层**：方法论决定了规则，规则决定了工作流，工作流决定了输出格式，输出格式决定了质量控制的方式
- **下层反馈上层**：实际使用中发现的问题，可以反过来优化上层的规则和方法论

---

## Wiki 和 Skill 的分工

| 层级 | 放哪里 | 说明 |
|---|---|---|
| 方法论层 | **Wiki** | 通用概念，跨项目复用，互相链接 |
| 执行规则层 | **Skill 内部** | 每个 Skill 有自己的具体规则 |
| 工作流层 | **Skill 内部** | 每个 Skill 有自己的交互流程 |
| 输出工具层 | **Skill 内部** | 每个 Skill 有自己的输出模板 |
| 质量控制层 | **Skill 内部** | 每个 Skill 有自己的验收标准 |

**Wiki 存第一层（方法论），Skill 存全部五层。**

---

## 判断一个 Skill 好不好

看五层是不是都齐全：
- ❌ 只有一坨 prompt → 不好用
- ✅ 有方法论 + 规则 + 工作流 + 输出模板 + 质量控制 → 好用

缺哪层补哪层。
