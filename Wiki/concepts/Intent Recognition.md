# Intent Recognition

意图识别。判断用户想要做什么，然后路由到对应的工作流或技能。

当一个 agent 集成了很多功能时，不能把所有 system prompt 都塞给一个 LLM——上下文太长、注意力分散、效果差。需要先识别意图，再调用对应的能力。

---

## 为什么需要意图识别

**问题：** 功能越多，system prompt 越长，LLM 越容易选错流程。
- 3 个功能 → 还能凑合
- 20 个功能 → 上下文爆炸，效果灾难

**解决方案：** 加一层意图识别。
```
用户输入 → 意图识别 → 路由到对应工作流/技能 → 执行
```

---

## 三种实现方法

### 1. LLM 识别
用轻量级 LLM（如 flash 版）输出意图，然后 switch case 路由。

- **优点**：能力最强，能理解复杂语义
- **缺点**：最慢，token 消耗大
- **适用**：复杂、模糊的用户输入

### 2. 正则/关键词匹配
代码层面匹配关键词或正则表达式。

- **优点**：最快、最稳、最好调试
- **缺点**：不够灵活，只能匹配预设模式
- **适用**：明确的、高频的指令

### 3. BERT/Rerank 模型
用语义相似度匹配，计算 query 和每个 intent 的相关度。

- **优点**：比 LLM 快，比正则灵活，可以微调
- **缺点**：能力不如 LLM
- **适用**：中等复杂度，有标注数据可以微调的场景

---

## 速度 vs 能力

| 方法 | 速度 | 能力 | 可微调 |
|---|---|---|---|
| 正则匹配 | 最快 | 最弱 | ❌ |
| BERT/Rerank | 中等 | 中等 | ✅ |
| LLM | 最慢 | 最强 | ✅ |

**实际使用：混合使用。** 先用正则快速拦截高频明确指令，剩下的交给 LLM 或 BERT。

---

## LLM 输出格式

用 LLM 做意图识别时，通常输出三个字段：

1. **intent** — 识别到的意图
2. **confidence** — 置信度（0-1）
3. **complement** — 补充信息（信息不够时，问用户要什么）

**示例：**
```json
{
  "intent": "申请请假",
  "confidence": 0.72,
  "complement": "请问您的工号是？"
}
```

---

## 低置信度处理

confidence 太低时，不要硬猜。

**处理方式：**
- 直接拒绝，让用户重新描述
- 或者问一个澄清问题
- 不要用低置信度的结果直接执行

**核心原则：** 猜错的代价比问一句大得多。

---

## 不只是第一次路由

意图识别不是只有"用户第一次输入时"那一下。

**贯穿整个交互过程：**
- 用户说"改一下这句" → 识别为"点式修改"意图 → 走修改流程
- 用户说"不行" → 识别为"否定"意图 → 走否定流程（问一个二元问题）
- 用户说"短一点" → 识别为"删减长度"意图 → 走删减流程
- 用户说"我有个想法" → 识别为"新想法"意图 → 走想法验证流程

每一轮用户输入，都要先识别意图，再走对应的工作流。

---

## 和 Skills 的关系

技能选择本质上就是意图识别。

通用 agent（如 Claude Code、OpenCode）里看不到显式的"意图识别层"，因为它已经整合到 skills 选取里了：
1. 用户输入
2. 读 SKILLS.md 索引
3. 找到最匹配的 skill
4. 注入 skill 内容 → 变成专业 agent

整个过程就是意图识别。

---

## 设计要点

1. **意图要互斥** — 两个意图不要太像，不然 LLM 也分不清
2. **数量要控制** — 不要一下搞 50 个意图，10-20 个比较合适
3. **要有兜底** — 识别不出来时怎么办？拒绝？转人工？还是走通用对话？
4. **可以迭代** — 收集识别错的 case，微调模型或补充正则
5. **信息不够就问** — 不要硬猜，补全信息再执行

---

## 不仅适用于 Agent

任何有多个功能入口的系统，都需要意图识别：
- 客服机器人（咨询/投诉/下单）
- 语音助手（打电话/发短信/设闹钟）
- 设计工具（生成/修改/导出）
- 剧本写作（写场景/改台词/剪长度）

本质都是：先判断用户要什么，再用对应的方式处理。
