# 上下文汇总与写入规则

## 一、上下文读取顺序

1. 今日日记是否已存在（存在则走轻量调整模式）。
2. 昨日日记中的未完成任务（`- [ ]`）。
3. 昨日日记 `## 明日计划`（若存在）。
4. `{{metaPaths.memory}}/STATUS.md` 的“本周待处理”。
5. 上周周记（若存在），并读取其中 `## 下周计划` 作为本周计划基线。
6. 本周计划进展（从上周周记 `## 下周计划` 与本周已执行任务提取，若存在）。
7. 本月月记（若存在）。
8. 活跃项目总览：`{{notePaths.activeProjects}}/📍 项目总览.md`（若存在）。

## 二、候选任务合并规则

- 先按来源分组展示，再让用户选择导入。
- 昨日日记 `## 明日计划` 作为优先候选，先向用户确认“沿用或调整”。
- 用户没选中的候选项不写入今日任务。
- 来源缺失时直接跳过，不报错，不展示空分组。
- 项目来源仅提供“今日可推进的活跃项目动作”，不直接塞入最终任务。
- 若存在“本周计划进展”，在候选任务后追加一段进展摘要（进行中/已完成/有风险）。

## 三、昨日计划确认规则

- 固定提问：`你昨天制定了以下计划，这些任务是否需要修改或补充？`
- 用户回复“不需要/不用改” -> 默认沿用昨日 `## 明日计划`。
- 用户回复具体修改 -> 以用户输入增删改后写入今日 `## 任务`。

## 四、今日聚焦规则

- 只确定 1 件“今天最重要的事”。
- 允许 AI 基于周/月方向给出候选，但最终以用户选择为准。
- 聚焦内容应具体、可执行、可在当天推进。

## 五、任务写入规则

- 今日聚焦写在任务首位并高亮。
- 若来自昨日 `## 明日计划`，先写入今日任务，再由用户确认保留 1-3 条重点项。
- 其他任务控制在 2-5 条，避免过载。
- 若任务来自“本周待处理”并被导入，当次会话应从 `STATUS.md` 删除对应条目。

## 六、文件更新规则

更新目标：`{{notePaths.dailyNotes}}/{{YYYY-MM-DD}}.md`

若文件已存在（例如白天 `ah-capture` 已写入内容，或用户手动维护过），采用“增删改”而非整段覆盖。

最小写入内容：

1. `## 今天最重要的事`
2. `## 任务`
3. 保留 `## 记录` 与 `## 晚间回顾` 占位
4. 保留 `## 明日计划` 占位

## 七、完成确认模板

```text

- 今日聚焦：{{focus}}
- 任务数：{{task_count}}
- 本周计划进展：{{week_progress}}
- 文件：{{notePaths.dailyNotes}}/{{YYYY-MM-DD}}.md

{{如果触发回顾提醒}}
💡 今日可执行：{{回顾类型}}（使用 {{命令}}）
```
