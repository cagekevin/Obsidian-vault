# 费曼风格护栏（Feynman Style Rubric）

> **吸收来源**：skills.sh `lijigang/ljg-skills@ljg-paper`（1,335 安装量）
> **适用场景**：当 `preferences.writing_style == "feynman"` 时，对 `summary_card` 的所有自然语言字段（method / contributions / limitations / recommended_questions[*].why）和 `deep_dive_answers[*].answer` 做风格重写。
> **核心主张**：读论文不是做学术，是猎取思想。默认 `analytical` 模式保留现状（信息密度优先）；开启 `feynman` 后切换为"活人在讲"的表达方式。

## 一、9 条红线（每条必过）

生成文本**写完后**逐条自检，任一条不过回改：

| # | 红线 | 判定标准 | 典型违例 → 正例 |
|:-:|---|---|---|
| 1 | **口语检验** | 你会这样跟朋友介绍一篇论文吗？不会 → 改 | ❌「本文提出了一种新颖的框架…」 → ✅「他们做了个东西，关键在…」 |
| 2 | **零术语** | 先用大白话落地，再顺带提术语名；必须用原文术语才能解释 = 还没懂 | ❌「通过对比学习机制…」 → ✅「让模型自己比两张图像不像不像（术语叫 contrastive learning）」 |
| 3 | **短词优先** | 能用两个字说的不用四个字 | ❌「本文提出了一种新的框架」 → ✅「他们做了个东西」 |
| 4 | **一句一事** | 每句只推一步逻辑 | ❌「我们提出方法并证明它在多个基准上效果好且速度快」 → ✅拆三句 |
| 5 | **具体** | 名词看得见，动词有力气；形容词能砍就砍 | ❌「显著的提升」 → ✅「ROUGE-L 从 34.6 涨到 46.2」 |
| 6 | **开头给理由** | 问题部分第一句让人想知道答案 | ❌「近年来 LLM 发展迅速」 → ✅「大模型明明很聪明，为什么一问具体事实就开始胡说？」 |
| 7 | **不填充** | 删学术套话（「近年来随着…的发展」「值得注意的是」「在某种程度上」） | — |
| 8 | **信任读者** | 说一遍够了，不重复结论 | — |
| 9 | **诚实** | 论文有硬伤就说有硬伤；看不懂的部分说看不懂 | ❌「本文贡献显著」← 其实方法与 baseline 差距不到 1% → ✅「提升其实很有限（< 1%），但思路值得留意」 |

**自检脚本化**：
- 红线 7 过滤词表：`近年来 | 随着...发展 | 值得注意的是 | 值得一提 | 在某种程度上 | 不言而喻 | 众所周知 | 本文提出了一种 | 具有重要意义 | 意义重大`
- 红线 2 触发词：文本中含 ≥ 3 个英文技术术语且无一句中文解释 → 触发改写

## 二、4 条写作原则

| # | 原则 | 判定 | 实施 |
|:-:|---|---|---|
| 1 | **一个锚点撑全文** | 找到一个具象的中心隐喻/场景/动作，所有概念围绕它生长 | skim 的 `method` 字段先给锚点比喻再讲机制；deep_dive 回答的第一句是锚点 |
| 2 | **推理外显** | 模拟"一个人想明白的过程"，让读者觉得差一步就是自己想到的 | 用「既然 A 是 B，那能不能 C 也是 D？」句式；避免直接给打包好的结论 |
| 3 | **变形替代定义** | 解释两个概念的关系时，把 A 连续变形成 B | ❌「LSTM 和 ResNet 是对偶的」→ ✅「把 LSTM 的循环展开 → 长得就像 ResNet」|
| 4 | **落点在能用** | 给出"这意味着你可以 ___" 而非"这让我们重新思考 ___" | recommended_questions[*].why / connection_points[*].insight 必须以动词型收尾 |

## 三、工具箱（选用，不强制）

讲解论文时可以拿的工具，**没有哪个是必须的**，按论文的特点选：

| 工具 | 适用场景 | 使用提示 |
|---|---|---|
| **类比** | 方法复杂但有结构 | 类比要"承重"——关键组件都能映射上，沿着类比走一遍方法 |
| **ASCII 图** | 组件关系 / 数据流 / 结构对比 | 读者先有概念脚手架再画，否则看不懂 |
| **餐巾纸速写** | 新旧对比 | 「以前这么想，现在应该这么想」并排 |
| **好问题** | 问题定义不明显 | 把论文解决的困境变成一个让外行好奇的问题 |
| **递进例子** | 概念抽象 | 从简单到复杂一步步搭 |
| **反问入链** | 有隐含假设 | 用问题打开 |

## 四、`writing_style` 切换协议

### 4.1 preferences 字段

```json
"preferences": {
  "writing_style": "analytical"   // "analytical"（默认）| "feynman"
}
```

### 4.2 各字段的风格差异对照

| 字段 | `analytical` 模式（默认） | `feynman` 模式 |
|---|---|---|
| `research_question` | 「能否用 LM 自生成指令媲美人工标注？」 | 「如果让模型自己给自己出题，结果会比人出的差吗？」 |
| `method` 首句 | 「用 175 条种子指令 bootstrap 迭代生成 52K 样本」 | 「从 175 道题起手，让 GPT-3 自己抄自己出新题，抄到第 52000 道为止」|
| `contributions[0]` | 「提出 Self-Instruct 框架，首次证明 LM 可用自生成数据做指令微调」 | 「第一次证明：LM 真的可以当自己的老师」|
| `recommended_questions[0].why` | 「方法的可信度决定复现价值」 | 「你要是想复现，这步没搞懂就白搭」|
| `deep_dive_answers[0].answer` 首句 | 「作者采用 bootstrap 采样（p.5）」 | 「简单说：给个开头让模型自己往下编」|

### 4.3 模式混合规则

- **永不混用**：同一次 run 内所有自然语言字段必须统一风格
- **Provenance 字段不变**：`page` / `section` / `excerpt` 字段不做任何风格处理（忠实原文）
- **数字不变**：`key_results` 中的数字必须原样保留（费曼化只能包装周围文字，不能改数字）

## 五、验收（AI 自检 8 条）

生成 feynman 模式输出后，逐条过：

- [ ] 问题勾人：让不懂的人也想知道答案（红线 6）
- [ ] 有锚点：method / deep_dive 的首句给了一个具象的中心比喻（原则 1）
- [ ] 带着推：读者能感受到"一步步想明白"的过程，而非接收打包好的结论（原则 2）
- [ ] 外行能跟：不懂这个领域的聪明人读完能复述核心思路（红线 2）
- [ ] 启发能动手：`recommended_questions[*].why` 和 `connection_points[*].insight` 落点是"你可以 ___"（原则 4）
- [ ] 零割裂感：读完像一个人在跟你说「我读了篇论文，发现了个有意思的事」
- [ ] 数字未改：`key_results` 中的数字、页码、专有名词全部与原文一致
- [ ] 填充词为 0：扫描过滤词表（红线 7）后无命中

任一项不过 → 重写受影响字段，**不得回退到 analytical 风格凑数**。

## 六、常见错误（NEVER）

- **NEVER** 在 feynman 模式下保留"近年来随着...的发展" / "值得注意的是" 这类套话
- **NEVER** 把 `key_results` 的数字改成模糊表达（"效果显著提升"替代"+11.6 ROUGE"）
- **NEVER** 为了口语化把专有名词错译（LSTM / ResNet / Transformer 保留原名）
- **NEVER** 混合 analytical 和 feynman（同一 run 内要么全 analytical 要么全 feynman）
- **NEVER** 把 feynman 风格污染到 provenance 字段（page / section / excerpt 必须原文）
- **NEVER** 用过度的类比掩盖"没看懂"（红线 9 = 诚实；看不懂就说看不懂）
