# 迭代模式工作流

[触发条件]
    用户在已有 storyboard.md 的项目里提出修改 / 新增 / 调整 / 替换 / 风格变更。

[工作流]
    接住需求 → 判断变更深度 → 按深度追问 → 冲突检测 → 更新 spec → 告诉用户

[顶层规则]
    - 无缝衔接，不开场白，不重新跑 Phase 1-5
    - 直接读现有 spec，接住需求，按变更深度追问
    - 不打断用户工作流
    - 组合变更（一次包含多个层级）按最重的处理

---

## 重度变更

[触发条件]
    涉及 Phase 1 必问项变更 / 整体节奏类型变更 / 受众或核心信息变更。

[硬指标]
    能回答"这个变更怎么影响整支视频"——为什么改 / 影响哪些场景 / 整体节奏怎么变 / 原素材还能用吗 / 核心信息要不要跟着改。

[围栏]
    - 不接受"全部重做"——这是 0-1 模式，应该归档旧 spec
    - 必须二次确认（先生成"预计改动总览"让用户确认 OK 再动 spec 文件）

[完成后衔接文案]
    ✓ "改动总览我先列出来——[受影响 Scene 列表 + 整体节奏变化 + 素材增删]。你确认 OK 我再动 spec 文件。"

---

## 中度变更

[触发条件]
    加新场景 / 删场景 / 替换组件 / 大段重排。

[硬指标]
    能回答"具体改成什么样 + 与前后镜头怎么衔接"——改哪个 Scene / 改成什么组件 / 文案 / 时长 / 转场要不要跟着改。

[围栏]
    - 不允许悄悄改其他 Scene（只改用户指定的）
    - 时长变了必须重新核对总时长与节奏档位
    - 用户卡住时给 2-3 个具体方案 + 优劣，不开放问

[完成后衔接文案]
    ✓ "Scene 3 换成 `aroll.timeline-card`，时长从 2s 拉到 3s，前一镜头转场改成淡入。总时长 +1s。这样改我去动文件。"

---

## 轻度变更

[触发条件]
    调整旁白 / 屏显文案 / 改 accent 色 / 改装饰层 / 时长 ±0.5s 内。

[硬指标]
    确认理解正确——改什么 / 改成什么。

[围栏]
    - 不优化用户已确认的文案（除非明确要求）
    - 不改 markdown 结构

[完成后衔接文案]
    ✓ "accent 色从 #FF6A00 改成 #FF8C42，其他不动。"

---

## 微调指令集（模糊反馈 → 精确操作）

用户提的修改需求经常是模糊感受（"节奏有点平"、"画面不够强"），这类反馈**不是轻度或中度变更**——用户不是在指定改什么，而是在表达感受，需要 AI 自己诊断并动手。

[核心规则]
    - 用户给出模糊感受 → AI 不追问"能具体点吗"，而是自己做诊断 + 给出修改方案让用户确认
    - 用户确认后直接改，不需要再问细节

[感受 → 操作映射]

| 用户说 | AI 诊断什么 | 执行什么操作 |
|--------|------------|------------|
| "节奏有点平" | 扫描所有 Scene 的时长，看是否连续同速 ≥ 3 镜；最长与最短时长差 < 2 倍 | 选 2-3 个中段镜头缩短或拉长 30-50%，制造快慢对比；如果全片同速，在 hook段加快、中段插一个短镜 |
| "画面感不够强" | 检查画面描述中是否有小于 15 字的 Scene；检查景别是否连续中景固定 ≥ 2 镜 | 补画面描述细节（增加视觉元素、布局说明）；插一个特写或全景 reset |
| "开头不够抓人" | 检查第一镜：时长是否 > 2s？类型是否 hook？ | 缩短到 2s 内；如果第一镜是标题卡，换成 hook 型组件（大字/数字/反白闪屏） |
| "中间有点拖" | 找到时长远超平均值的 2-3 个镜头 | 砍 1 个或缩短到平均值；如果该段是教程型讲解，拆成 2 个快镜 |
| "结尾没力" | 检查最后一镜：是否有大字收束 / CTA / fade-out？ | 补大字 takeaway 或 CTA，最后 2s fade-out 到黑 |
| "感觉都一样" | 检查组件多样性：同一组件 ID 是否 ≥ 4 次？同一展现对象是否 ≥ 50%？ | 替换 2-3 个重复组件的 Scene 为其他类型 |
| "第 X 镜不够具体" | 读该 Scene 的画面描述，检查是否 < 20 字或只写了布局没写视觉元素 | 补视觉细节（颜色/形状/光线/材质/运动） |

[执行流程]
    用户表达感受 → AI 对照上表诊断 → 给出 1-3 处具体修改点让用户确认：
    "我觉得是这三个问题：[具体点]。我这么改你看行不行？"
    用户确认 ✓ → 直接改 spec，不用再问细节。

## [冲突检测]

加载现有 storyboard.md，新需求如果与现有内容冲突，必须直接指出 + 给方案让用户选。

[常见冲突类型]
    - 节奏冲突（总时长超 / 不够）
    - 组件冲突（与已选场景类型组合不符）
    - 素材冲突（依赖的素材原 spec 没有）
    - 转场冲突（触发硬规则，如反白闪屏 ≤ 2 次、溶解 / 划像每 3 Scene ≤ 1 次）
    - 核心信息冲突（新内容稀释 takeaway）

[停止追问的标准]
    - 能直接动手改 storyboard.md，不需要再猜或假设
    - 改完之后用户不会说"不是这个意思"

---

## [更新文档]

[纪律]
    - 在原 spec 上精确修改（不另存新文件）
    - 不改文档 markdown 结构（标题层级 / 列表风格）
    - 不删除用户的注释 / 备注
    - 改了分镜表 → 用 `references/spec-rules.md` 的字段约束和 [交付前自检清单] 重新核对
      （组件 ID 真实性 / 总时长误差 / 反白闪屏额度 / 转场字段齐全）

---

## [完成后告诉用户]

    storyboard.md 已更新。
    - 改动总览：[哪些 Scene / 总时长怎么变 / 素材新增项]
    - 如果影响渲染，重新提交 storyboard.md 生成视频
