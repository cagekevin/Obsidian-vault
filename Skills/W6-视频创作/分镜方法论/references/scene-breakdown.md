# 分镜拆解方法论

[使用时机]
    - 0-1 模式 Phase 1-5 都问完后，进入分镜起草阶段
    - 迭代模式加新场景 / 大段重排时

    [核心理念]
    分镜 = 把脚本翻译成可执行的画面 + 动效，不是切段落。
    每个 Scene 必须回答 5 个问题：
    1. 信息载荷 —— 这一镜让观众 get 到什么？
    2. 组件 ID —— 用 components-catalog 哪个组件承载？
    3. 时长 —— 几秒？
    4. 镜头语言 —— 怎么拍？（景别 + 运镜 + 机位）
    5. 转场 —— 怎么进、怎么出？

---

## [输入]

[输入形态]

    1. 逐字稿（最理想）
        已有完整旁白文本，按句号 / 语意单元切分

    2. 卖点列表
        3-7 个要点，每点要单独镜头表达

    3. 大纲
        三幕结构（开-中-收），每幕有粗略要点

    4. 都没有
        基于核心信息 + 受众 + 时长，你写草稿（前置：Phase 1 五维度齐全）

[输入校验]
    每种输入都必须能回答：
    - 总时长（来自 Phase 1）
    - 信息密度档位（hook / 教程 / 纪录）
    - 核心信息（≤ 12 字）
    - 叙事结构（来自 Phase 3.5 / 3.6 / 3.7）：
      节拍曲线 + 情绪曲线 + 音画关系 + 同质化反例
      → 拆分镜前必须先读 storyboard.md 的 [叙事结构] 章节
      → 每个 Scene 都要落到节拍曲线的某一段

---

## [六步法]

### [一、内容分割]

    [逐字稿]
        1. 按句号断句 → 候选镜头列表
        2. 合并连续短句（< 8 字）→ 节奏镜头
            例："来。看这个。这就是答案。" → 合并为 1 镜，但保留 3 段动效层级
        3. 长句（> 30 字）拆成多镜头
            拆分点：连词（但是 / 因为 / 所以）/ 转折标记 / 数据出现处
        4. 排比句 / 列举句 → 每项独立镜头（视觉上做"序列感"）

    [卖点列表]
        每个卖点拆成：
        - 1 个核心镜头（陈述卖点）
        - 0-2 个支撑镜头（证据 / 数据 / 演示）
        - 总数控制：3-7 个卖点 × 平均 2 镜 = 6-14 个 Scene

    [大纲]
        三段分配：
        - 开（hook）：1-3 镜头，占 15-20%（30s 视频 → 5-6s 内 / 60s → 10-12s 内）
        - 中（核心）：5-15 镜头，占 60-70%（主要信息密度区，节奏快-慢交替）
        - 收（cta / 收束）：1-2 镜头，占 10-15%（收束核心信息 + 可选 CTA）

    [都没有]
        你根据 Phase 1 五维度起草大纲：
        - 推算镜头总数（按节奏类型 × 总时长 → 大致镜头数）
        - 起草大纲三段
        - 必须让用户确认大纲再继续拆细
        - 不要从零跳到逐字稿——中间要有大纲校准

### [二、节奏分配]

    [按节奏类型分配时长基准]

        - hook 型（每镜 0.8-2s 主，关键镜头 2-3s）
        - 教程型（每镜 2-5s 主，关键解释 5-7s）
        - 纪录型（每镜 3-8s 主，强调镜头 6-10s）

        详细分配见 pacing-rules.md。

    [总时长校验]
        - 所有镜头时长 + 转场时长（粗略） ≈ 用户设定的总时长
        - 转场粗估 0.2-0.5s/转场（精确值后续渲染时定）
        - 误差 ±0.5s 以内允许

    [节奏曲线纪律]
        - 不允许"全片均速"——会让观众疲劳
        - 推荐节奏曲线：快-快-慢-快-慢-慢-快收尾
        - 关键信息镜头前一镜可以加快（制造对比，凸显关键镜头的"停顿感"）

### [三、镜头打标]

    [标签集合（一镜可多标签）]

        - `[hook]` 抓注意力（开头 / 大转折 / 反差冲击）
        - `[data]` 数据 / 统计 / 排名
        - `[concept]` 概念 / 定义 / 抽象阐释
        - `[emotion]` 情绪 / 价值观 / 故事感
        - `[emotion-intensity=N]` 情绪强度量化，N=1-5（1=冷静淡漠，3=中性叙事，5=极致冲击）；只在关键 Scene 标注，不必每镜写
        - `[demo]` 演示 / 操作 / 产品功能
        - `[versus]` 对比 / 反差 / A vs B
        - `[structure]` 结构 / 层级 / 分类
        - `[flow]` 流程 / 步骤 / 序列
        - `[cta]` 行动召唤 / 引导动作
        - `[bridge]` 过渡 / 桥接（不承载主信息，但连接两段）
        - `[summary]` 总结 / 收束 / 重申核心信息

        **[catalog 专用标签]**
        根据当前 catalog 的 frontmatter `tags` 字段追加使用（如当前 catalog 定义了 mechanism、anatomy 等标签）。

    [打标纪律]
        - 每个 Scene 至少 1 个主标签 + 0-2 个副标签
        - `[bridge]` 占比 ≤ 15%（桥接太多会稀释信息密度）
        - `[hook]` 标签的镜头必须落在视频开头 3s 内（至少 1 个）
        - 一个 Scene 同时具备 `[data] + [concept]` → 警告"信息复合度高，注意信息过载"

### [四、组件匹配]

    [按标签 + 内容性质从 components-catalog 选组件]

    | 标签 | 匹配策略 | 参考 catalog 中的展现对象 |
    |------|---------|---------------------------|
    | hook | 选择视觉冲击力最强的大构图 | panorama / hero / bridge |
    | data | 选择以数据或数字为核心的镜头 | data / scale |
    | concept | 选择以概念讲解为核心的镜头 | explain / label |
    | emotion | 选择以氛围烘托为主的镜头 | emotion / force |
    | demo | 选择操作或界面展示的镜头 | demo / mechanism |
    | versus | 选择对比或对照的镜头 | contrast |
    | structure | 选择层级或全局结构的镜头 | structure / panorama |
    | flow | 选择流程或链路的镜头 | flow / mechanism |
    | cta | 选择视觉收束感强的镜头 | hero / panorama |
    | bridge | 选择过渡或切换的镜头 | bridge |
    | summary | 选择简洁收束的画面 | label / hero |

    [catalog 动态匹配]
    具体标签→组件的映射，优先参考当前 catalog 中各条目的「何时用」字段。
    找不到匹配时，按标签名在 frontmatter 的 `shot_types` 中模糊查找。

    [组件匹配纪律]
        - 不允许"自创组件"——必须落到 components-catalog 已有 ID
        - 某 Scene 没有合适组件 → 重写文案 / 改标签，不要发明组件
        - 同一组件全片用 ≥ 4 次 → 警告"视觉单调"
        - 同一类组件（如同一展现对象类型）用 ≥ 5 次 → 警告"视觉过密集"

    [详细清单]
        见 components-catalog.md（含每个组件的适用场景 / 何时不用 / 内容期待）。

### [五、转场决策]

    每个 Scene 都要明确"转场进入"和"转场离开"，只写类型，不写参数。
    不允许 jump cut。

    转场类型（4 种粗粒度）：
    - 硬切（hard cut）—— 默认，瞬间切换
    - 软切（crossfade）—— 柔和过渡
    - 反白闪屏 —— 硬规则：≤ 2 次/视频，间隔 ≥ 8s
    - 溶解 —— 前镜渐隐、后镜渐显
    - 划像 —— 边界线扫过切换画面

    具体 ease / duration 由视觉富化阶段决定。

### [六、镜头语言决策]

    [核心理念]
        每镜的"怎么拍"决定观众怎么感受。同一个内容用特写拍和用全景拍，传达的信息完全不一样。

    [景别选择]
        景别决定观众和内容之间的心理距离：

        - **特写**（Close-up）：人脸 / 细节 / 关键数字。观众被迫注视，无路可逃。用于情绪爆发、关键数据、决定性瞬间。
        - **中景**（Medium）：人物半身 / 卡片 / 图表主体。最常见的景别，像正常对话距离。用于大多数叙事镜头。
        - **全景**（Full / Wide）：全身 / 全屏布局 / 场景全貌。观众看到整体。用于建立空间关系、展示规模、章节开篇。
        - **大远景**（Extreme Wide）：渺小的人 vs 广阔环境。用于史诗感、孤寂感、或反衬"这个系统有多大"。

        [景别纪律]
        - 特写 ≤ 全片 Scene 的 20%（太密集会窒息）
        - 同一景别连续 ≥ 3 镜 → 观众视觉疲劳，必须换景别
        - 关键信息句优先特写（让观众没地方躲）
        - 过渡 / 桥接镜推荐用中景或全景（降低认知负荷）

    [运镜选择]
        运镜决定画面是静态阅读还是动态推进：

        - **固定**（Static / Lock-off）：镜头不动，让内容自己说话。用于大字海报、数据图表、引用块。大部分 Scene 默认固定。
        - **推**（Push / Zoom-in）：镜头向主体推进，压缩空间，增强紧迫感。用于揭示细节、制造"靠近了"的感觉。
        - **拉**（Pull / Zoom-out）：镜头远离主体，扩展空间，产生疏离或释然。用于展示全貌、情绪收束。
        - **摇**（Pan）：镜头水平或垂直转动。用于连接两个元素、扫视场景。
        - **移**（Track / Dolly）：镜头平行于主体移动。用于并列展示、跟随动作、建立"我们一起走"的代入感。
        - **跟**（Follow）：镜头跟随移动主体。用于人物走动、产品运动、动态演示。

        [运镜纪律]
        - 固定镜头 ≥ 60%（太多运镜会让观众眩晕）
        - 运镜必须有动机（跟随主体 / 揭示信息 / 改变情绪），不为运镜而运镜
        - 推 / 拉 / 摇 / 移 全片各 ≤ 3 次（否则显廉价）

    [机位选择]

        - **平视**（Eye Level）：最自然，观众代入感最强。默认机位。
        - **仰角**（Low Angle）：主体显得强大 / 威严 / 不可挑战。用于 hero 镜头、权威引用。
        - **俯角**（High Angle）：主体显得渺小 / 脆弱 / 一览无余。用于俯瞰数据、解释系统结构。
        - **POV**（Point of View）：模拟主体视角。观众"变成"角色。用于沉浸体验、第一人称叙事。
        - **过肩**（Over Shoulder）：对话或多角色场景，加强"有人在看"的存在感。

    [镜头语言与组件匹配参考]

    | 展现对象类型 | 推荐景别 | 推荐运镜 | 推荐机位 |
    |---------|---------|---------|---------|
    | hero / panorama | 全景 | 固定 / 推 | 平视 |
    | data / scale | 中景→特写 | 推（揭示细节） | 平视 |
    | explain / label | 中景 | 固定 / 摇 | 平视 |
    | flow / mechanism | 全景→中景 | 移（沿流程方向） | 俯角 |
    | demo | 特写 | 固定 / 推 | 平视 |
    | overlay | 特写 / 中景 | 固定 | 平视 |
    | contrast / structure | 中景→全景 | 拉（展示全貌） | 俯角 |
    | illustration | 全景 | 固定 | 平视 |

    [常见错误]
        - 全片所有 Scene 都是中景固定平视 → 视觉叙事死亡，观众看 30s 就累
        - 连续 3 镜特写 → 观众想呼吸，必须插一个全景或中景 reset
        - 运镜无动机 → "推"了但没新信息揭示 → 眩晕 + 廉价
        - 全文无视 9:16 / 16:9 差异 → 竖屏用全景人物太小，横屏用特写太满

### [七、素材绑定]

    [每镜必须绑定的素材]

        - 旁白文案
            - 从逐字稿对应段落抽取
            - 如无逐字稿，你根据 Scene 主题 + 屏显文案补
            - 用 TTS → 在素材依赖里标 "voice/scene-XX.mp3"

        - 屏显文案
            - 旁白的关键词（不是全文！）
            - 12 字以内的核心句
            - 高亮词单独标记

        - 素材依赖
            标注需要的：图片（路径）/ 视频（路径 + 时长 + 起始偏移）/
            数据文件（data.json）/ Lottie / 3D 模型（路径 + camera 参数）/ 非默认字体

        - 缺素材的镜头
            标记 `[待补充素材]` + 退化方案
            例：`[待补充素材] 真人产品演示视频；退化方案：用 demo 类型组件模拟产品界面`

    [素材绑定纪律]
        - 不允许"过度乐观"——没素材就标待补充，不要默认你能生成
        - 素材路径用相对路径（./assets/xxx）
        - 同一素材在多 Scene 复用 → 明确标注 "复用自 Scene XX"

## [输出格式]

[每镜必填字段]
    严格对齐 templates/storyboard-template.md「Script · 分镜表」节的字段，顺序不变：

    标题行：Scene NN · [start]s – [end]s · [节拍标签，如 hook / 反差 / 数据 / 高潮 / CTA]

    13 个内容字段：
        1.  类型（按当前 catalog 的定义填写——如 主线讲解 / 辅助画面 / 过场转场）
        2.  组件（来自 components-catalog.md 的真实 ID）
        3.  旁白文案（完整原文 / —）
        4.  屏显文案（屏幕显示的文字 / —）
        5.  期待内容（本镜传达的内容 / 数据 / 概念 / 情绪）
        6.  期待效果（观众应该产生的反应：震撼 / 理解 / 记住 / 笑 / 共鸣）
        7.  画面描述（镜头里有什么、布局、是否 3D、关键视觉要素）
        8.  镜头语言（景别 + 运镜 + 机位，三项必须全写）
        9.  动效要点（动词级，如 SLAMS / CASCADE / floats）
        10. 音效描述（无音效写「无」；有写具体类型 + 时间点）
        11. 转场进入 + 转场离开（类型级：硬切 / 溶解 / 划像 / 淡入淡出 / 反白闪屏）
        12. 素材依赖（narration.wav 段 / bgm.mp3 / 待搜索 X / 已有素材 Y）
        13. 参考画面（参考视频时间码 / 参考图链接 / 草图；可选）

[Scene 排版]

```
Scene XX  [start-end / duration]  · [节拍标签]
├─ 类型：主线讲解 / 辅助画面 / 过场转场（按当前 catalog 分类）
├─ 组件 ID：broll-hero.big-number
├─ 旁白文案：…
├─ 屏显文案：…（含高亮词标记）
├─ [期待内容]：本镜要传达的具体内容（数据/概念/情绪/动作）
├─ [期待效果]：本镜应该让观众产生什么反应（震撼/理解/记住/笑/共鸣）
├─ 画面描述：（画面构成 / 布局 / 是否 3D / 装饰层）
├─ 镜头语言：特写 · 固定 · 平视
├─ 动效要点：（关键入场 / 出场 / 内部动效，动词级）
├─ 音效描述：（无 / 具体类型 + 时间点，如「3.0s · 短促 click，volume 0.3」）
├─ 转场进入 ← / 转场离开 →
├─ 素材依赖：（路径 / [待补充素材]）
└─ 参考画面：（时间码 / 参考链接 / 草图）
```

[期待内容] 和 [期待效果] 是产品需求——渲染时按这两个字段调整视觉强度、节奏、装饰层。
[音效描述] 字段不能省，无音效也要明写「无」，避免输出时被遗漏。

[音效设计原则]
    音效不是"加上去的"，是从镜头信息载荷里长出来的。

    [音效的三大职责]
        - **叙事音效**：推动叙事、强调信息（标注弹出 blip、数据 tick、大字砸入 thump）。最常用，每个 Scene 几乎都需要至少一处。
        - **氛围音效**：建立环境感（低频嗡鸣、空间混响、远距离的风声）。用于沉浸镜头、留白段落、过渡桥接。
        - **节奏音效**：卡节拍的冲击音（鼓点+反白闪屏、重音+大字 SLAMS）。用于 hook 镜头、高潮段、CTA 收尾。

    [音效设计纪律]
        - 每个 Scene 至少 1 个音效触发点（无音效也必须明写「无」——空本身就是一种音效设计）
        - 同一音效全片复用 ≤ 6 次（否则产生模板感）
        - 音效音量与信息强度成正比：
            - 关键信息（hook / 数据 / 转折）→ volume 0.4-0.7
            - 日常叙事 / 过渡 → volume 0.2-0.35
            - 氛围底音 → volume 0.1-0.2
        - 旁白出现时音效 ducking（压低）到 0.15，不给旁白让道的音效不如不加
        - 反白闪屏的"静音瞬间"是最强的音效——不要往里塞杂音
        - 末镜 fade-out 前 1-2s 停止所有音效，只留 BGM fade

    [音效与情绪强度对照]

    | 情绪强度 | 音效策略 | 例子 |
    |---------|---------|------|
    | 1-2（冷静） | 无音效或极轻氛围音 | 纪录片旁白段 |
    | 3（中性） | 信息音效（blip / tick） | 标注弹出 |
    | 4（强） | 冲击音效（thump / whoosh） | 大字砸入 |
    | 5（极致） | 冲击音 + 静音对比 | 筷子臂合拢→0.3s 全静音 |

[动效要点]
    只写动词级别（choreography verbs），不写具体动效参数。
    常用动词：SLAMS / CASCADE / FLOATS / WHIPS / DISSOLVES / PULSES / SPIRALS / SCRATCHES。

    合格："title SLAMS 入场 + subtitle CASCADE 错峰"
    反例："y 30→0, 700ms ease power3.out"（这是视觉富化的活儿）

    具体 ease / duration / 物理参数后续富化时再写。

## [自检清单]

[输出后逐项检查]

```
□ 每个 Scene 都锚定到 components-catalog 里的组件 ID（没有自创）
□ 总时长 = 所有 Scene 时长 + 转场时长（误差 ±0.5s 以内）
□ 反白闪屏 ≤ 2 次（全片硬上限）
```

## [常见陷阱]

[陷阱 1：按句号切完就当分镜]
    问题：句号不等于视觉切点
    应对：再过一遍，看哪些短句要合并，哪些长句要拆分

[陷阱 2：每个 Scene 都用大字 hero]
    问题：视觉单调，观众疲劳
    应对：组件多样性自检——同一组件 ≤ 4 次

[陷阱 3：转场字段写具体参数 / 全用同一种]
    问题：写出 "crossfade 300ms ease-out" → 越界
    应对：只写类型（硬切 / 溶解 / 划像 / 淡入淡出 / 反白闪屏）

[陷阱 4：反白闪屏每个 Scene 都用]
    问题：违反 hard rule（≤ 2 次）
    应对：留给最关键的两个转折点（一般在 hook 进入 + 数据高潮）

[陷阱 5：节奏全片均速]
    问题：观众没记忆点，所有镜头一样长
    应对：检查节奏曲线，必须有快慢对比

[陷阱 6：所有 Scene 都堆满信息]
    问题：信息密度过载，观众消化不了
    应对：穿插 [bridge] 或 [emotion] 镜头给观众喘息（但占比 ≤ 15%）

[陷阱 7：忘记给数据 Scene 标数据源]
    问题：渲染时无数据可用
    应对：data Scene 必须标 data.json 路径或 inline 数据

[陷阱 8：讲解镜头旁白不全]
    问题：字幕高亮无法精确同步
    应对：包含逐词字幕的镜头必须有完整旁白文本（不能用"省略号"）

[陷阱 9：全片景别单调]
    问题：所有 Scene 都是中景固定平视，视觉叙事死亡
    应对：检查景别多样性——插入全景 reset 或特写冲击

[陷阱 10：运镜无动机]
    问题：为运镜而运镜，推/拉/摇没有揭示新信息
    应对：每次运镜问"这个动作让观众看到了什么之前看不到的？"

[陷阱 11：音效堆砌]
    问题：每个 Scene 塞 3-5 个音效，信息被淹没
    应对：全片选 2-3 个核心音效反复使用（blip / tick / thump），其余砍掉

[陷阱 12：缺参考画面]
    问题：画面描述写"科技感"，观众和 AI 各想各的
    应对：给具体参考链接或截图帧，比 50 字描述精确 10 倍
