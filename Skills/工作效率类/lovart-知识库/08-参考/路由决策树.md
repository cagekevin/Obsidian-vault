我来分享我的意图路由判断逻辑。这确实是一个复杂的决策系统，我会从**信号识别 → 任务分类 → 路由决策**三个层面来讲解。
---
## 一、核心判断维度
| 维度 | 关键信号 | 路由方向 |
|------|---------|---------|
| **动作动词** | 画/生成/创建 → 新建；改/修/换 → 编辑；合/拼/剪 → 合成 | 决定主工具类型 |
| **对象类型** | 图/视频/海报/PPT/HTML | 决定输出格式 |
| **输入状态** | 有无上传文件、参考图、品牌Kit | 决定是0→1生成还是基于素材 |
| **复杂度** | 单场景 vs 多场景、有无分镜/故事线 | 决定是否需子代理规划 |
| **约束条件** | 尺寸、风格参考、品牌规范 | 决定参数填充策略 |
---
## 二、典型场景的分流规则
### 场景1：「画一张图」类请求
```
信号提取：
├── 动词：画/生成/创建/做一张
├── 对象：图/图片/海报/插画
├── 输入：无上传文件 或 仅有风格参考图
└── 复杂度：单一场景描述
路由决策：
→ generate_media (task_type=generate_image)
→ 检查是否有尺寸要求（target_size/aspect_ratio）
→ 检查是否有参考图（reference_image_urls）
```
**关键判断**：无 `source_image_url`（没有要编辑的基底图），走生成而非编辑。
---
### 场景2：「做个品牌规范」类请求
```
信号提取：
├── 关键词：品牌规范/Brand Kit/VI手册/品牌指南
├── 输出形式：PDF文档 或 多页展示
├── 需要内容：Logo、色彩、字体、应用场景
└── 复杂度：多页、结构化
路由决策：
→ 先调用 asset_factory 检索现有品牌资产
→ 再调用 generate_slides 生成品牌规范PPT
→ 或调用 generate_media 生成品牌应用场景图集
```
**关键判断**：涉及「Kit」概念时，必须先检索资产，再决定生成策略。
---
### 场景3：「帮我改一下」类请求
```
信号提取：
├── 动词：改/修/换/调整/去掉/加上
├── 输入：必有上传文件（source_image_url）
├── 修改范围：局部（<50%）vs 全局
└── 修改类型：内容替换 vs 风格迁移 vs 尺寸调整
路由决策：
├── 局部修改（换背景、去物体、改颜色）→ edit_media (edit_image)
├── 风格重绘（保持构图换风格）→ generate_media (source_image_url + reference_image_urls)
├── 仅改尺寸/比例 → edit_media (reframe)
└── 放大分辨率 → edit_media (upscale)
```
**关键判断**：必须有 `source_image_url`，且根据修改范围选择 edit 还是 generate。

## 三、复杂意图的拆解规则
### 当用户一句话包含多个动作时：
| 用户表述 | 拆解 | 执行顺序 |
|---------|------|---------|
| "先搜一下参考，然后画出来" | 搜索 → 生成 | grounding_hub → generate_media |
| "按这个风格改一下这张图" | 风格参考 + 编辑 | generate_media (source + reference) |
| "做个视频，要有分镜" | 故事板规划 → 分镜生成 | call_creation_subagent(visual_planner) → 逐帧生成 |
| "把这几张图拼成视频" | 素材整理 → 时间线合成 | edit_video_ffmpeg |

## 四、我的决策树（简化版）
```
开始
  │
  ▼
是否有明确动作动词？
  ├── 否 → 请求澄清意图
  │
  └── 是 → 判断动作类型
            │
            ├── 新建类（画/生成/创建）
            │     ├── 有分镜/多场景需求？→ visual_planner
            │     ├── 是PPT/幻灯片？→ generate_slides
            │     ├── 是HTML页面？→ generate_html
            │     └── 单图/单视频 → generate_media
            │
            ├── 编辑类（改/修/换）
            │     ├── 有上传文件？→ 确定 source_image_url
            │     │     ├── 局部修改 → edit_media (edit_image/edit_video)
            │     │     └── 全局重绘 → generate_media (img2img)
            │     └── 无上传文件 → 请求提供素材
            │
            ├── 合成类（拼/剪/合）
            │     └── edit_video_ffmpeg / synthesize_media
            │
            └── 检索类（搜/找/查）
                  └── grounding_hub / asset_factory
```

## 五、给你的系统设计的建议
1. **优先级规则**：显式动词 > 隐式上下文 > 默认行为
2. **歧义处理**：当多个工具都「可能匹配」时，优先选择**约束更多**的路径（比如有上传文件时优先走编辑而非生成）
3. **用户确认**：复杂工作流（如先规划再生成）在执行前展示计划，等待确认
4. **失败回退**：工具调用失败后，提供替代方案而非直接报错
