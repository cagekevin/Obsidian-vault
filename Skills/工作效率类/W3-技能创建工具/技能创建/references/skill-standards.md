# 技能规范检查清单 (Skill Standards)

本文件定义了技能文件应遵守的各项标准，供审计时动态加载逐项对照。

## A — 结构规范

### A1 文件与目录
- 前缀：工程类 `S`，工作效率类 `W`
- 目录单层扁平，禁止多层嵌套（`图片/W7-图片/` ❌ → `W7-图片/` ✅）
- 独立技能文件夹名 = 主文件名（如 `W8-格式化/W8-格式化.md`）
- **Hub 独占编号，子文件零编号（红线⚠）**：枢纽入口文件独占该编号（如 `W1-沟通.md`），其下分支文件夹及文件名禁止出现任何 S/W 编号前缀。例如 `W1-沟通/W2-整理上下文/` ❌ → `W1-沟通/整理上下文/` ✅。否则 skills-init 会将分支误注册为独立技能，造成编号冲突。
- 子技能无编号，挂在枢纽下通过 `<route-table>` 分流
- **子技能独立目录（红线⚠）**：每个子技能拥有独立文件夹，主文件 + 自身`references/` 放在其内，禁止平铺堆在 Hub 同级。如 `ecommerce-image/ecommerce-image.md` ✅，而非 `ecommerce-image.md` ❌。Hub 自身文件仍保持在桶目录根级。

### A2 Frontmatter
- `name`：纯中文，不含编号前缀
- `description`：首句核心功能描述，末句 `Use when [触发场景].` 封底
- `metadata.pattern`：标注 5 大设计模式之一（tool-wrapper / generator / reviewer / inversion / pipeline）
- `metadata.category`（可选）：分类标签，用于编号用完后按分类路由。建议值：开发 / 审计 / 设计 / 沟通 / 通用。已有编号的技能可不加，新建技能建议添加。

### A3 核心标签
- 必须包含 `<what-to-do>` 和 `<supporting-info>` 区块
- 主流程内容放入 `<what-to-do>`，参考材料放入 `<supporting-info>`

## B — 内容一致性

### B1 变量命名
- 同一概念在全文中使用相同名称
- Rules 区与正文中对同一规则的表述名称一致

### B2 逻辑通顺
- 两条规则不得自相矛盾
- 前后定义一致

### B3 引用完整性
- 文件内引用的子文件/路径物理存在
- `<route-table>` 中声明的文件在磁盘上存在
- 磁盘上的文件没有遗漏注册到路由表

### B3.5 路由表规范
- Hub 文件必须包含 `<route-table>`，每行格式：`| 路由 | 文件 | 触发关键词 |`
- 每行首列为路由名（中文），次列为文件相对路径，末列为逗号分隔的触发关键词
- 分支文件在磁盘上必须有对应的物理文件
- Hub 自身的 description 必须包含所有分支路由的名称，以便 AI 通过关键词匹配触发
- 子文件（分支）无需也不应有独立的编号前缀

### B4 重复与冗余
- 同一事项不在多个位置以不同表述各说一遍

### B5 Router 完整性
- 分支文件全部声明在路由表中
- 无声明了但不存在的问题

### B6 Trigger 完整性
- 声明了双模触发（MODE A / MODE B），两种模式都实现
- 不存在声明了但没实现逻辑的分支

### B7 Example 对齐
- End-to-End Example 的步骤/参数数量与正文 Checklist 一致

### B8 种子文件
- `<supporting-info>` 引用的文件在目录中物理存在，路径正确

## C — 写作质量（新增）

### C1 语气与表达
- 以专家身份开头，如"你是 XX 领域专家，目标是 XX"，而非"本技能提供全面覆盖..."
- 用祈使句 + 主动语态（"做 X"优于"建议考虑 X"），严禁教材式/维基百科式语气
- 有观点 > 中立 —— 直接说哪个有效、哪个无效，不罗列选项让人自己选

### C2 解释原因
- **关键**：用"因为..."解释为什么这样做，代替硬邦邦的 ALWAYS/NEVER/必须
- 如果写了全大写的 MUST 或 NEVER，停下来问自己：能不能改成解释原因？
- 指令不依赖"死记硬背"，AI 理解了原因才能举一反三

### C3 精简
- SKILL.md 控制在 500 行 / 10KB 以内，超过则拆到 references/
- 删掉不干活的内容 —— 每行都要问"没有这一行会怎样？"
- 大型 reference 文件（>300行）必须带头目录

### C4 避免过拟合
- 不要让 skill 只对测试用例有效 —— 要能应对没见过的场景
- 如果某个问题反复出现，**不要加更多 MUST**，换一种不同的比喻/模式来重写
- 读测试日志（transcript）不只是看最终输出 —— 如果 skill 让 AI 浪费时间，删掉那部分

### C5 输出质量
- 定义了输出格式（模板 / 表格 / 清单 / 报告结构）
- 提供了 Input → Output 的对照示例
- 复杂判断给出了信心标记（🟢已验证 / 🟡中等 / 🔴假设）
- 输出制品表：将常见请求映射到具体可交付物（"当你要求... 你会得到..."）

### C6 交叉引用导航
- 末尾有关联技能列表，带**什么时候用**和**什么时候不用**的消歧义
- 交叉引用是双向的（A 提到 B，B 也要提到 A）

## D — 架构设计评价

### D1 模块边界
- 技能的职责边界清晰，不越界做不该做的事

### D2 Hub-Spoke 必要性
- 当前内容复杂度是否需要枢纽分流？一张 .md 能否装下？

### D3 同类重叠
- 和已有技能之间有无功能重叠？选择边界是否清楚？

### D4 流程设计
- 阶段/步骤划分自然，没有"为凑结构而硬分阶段"

## E — 评测质量（新增）

### E1 对照实验
- 是否同时跑了**带 skill** 和 **不带 skill（baseline）** 的对照？
- 如果两者结果差距不大，说明 skill 没创造价值，需要重写

### E2 迭代历史
- 至少经过了 1 轮以上的迭代（测→改→再测→再看）
- 每次迭代有独立的目录记录（iteration-1/、iteration-2/）

### E3 用户反馈
- 测试结果通过可视化报告展示给用户
- 用户看完后给出了反馈（哪些好、哪些不对）
- 反馈被实际用于下一轮改进

### E4 触发测试集
- 创建了 20 条 query 的触发测试集（10条应触发 + 10条不应触发）
- 不应触发的 query 选的是真正有迷惑性的**近义词陷阱**，不是明显不相关的
- query 真实具体（含文件路径、公司名、用户身份等细节）
- 每条 query 跑了至少 3 次统计触发率
- 选测试集分数最高的 description（不选训练集最高的，防过拟合）

### E5 重复工作检测
- 查看了测试日志中 AI 是否重复写了类似的 helper 脚本
- 如果是，打包到 `scripts/` 中统一调用
