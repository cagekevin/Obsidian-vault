# Skill 设计 SOP 框架

> 基于产品化思维的 Skill 设计标准作业流程，从定位到交付的完整方法论。

---

## 一、定位阶段：明确 Skill 要解决什么问题

这是设计 Skill 的第一步，核心是搞清楚「这个 Skill 到底解决谁的什么问题」。

### 1.1 核心问题

- 这个 Skill 解决的是哪类人的什么具体问题？
- 是一次性需求还是高频重复需求？
- 用户现在是怎么解决这个问题的？痛点有多痛？

### 1.2 输入与输出

| 类型 | 内容 |
|------|------|
| 输入 | 用户的模糊需求、痛点描述、使用场景 |
| 输出 | 一句话定位：「这是一个帮 [谁] 解决 [什么问题] 的 Skill」 |

### 1.3 检查清单

- [ ] 能用一句话说清楚这个 Skill 是干什么的
- [ ] 明确了目标用户是谁
- [ ] 找到了用户当前的替代方案和痛点
- [ ] 确认这是一个可被工具化的重复流程

### 1.4 正反示例

**❌ 反面：**「做一个设计 Skill」—— 太宽泛，不知道解决什么问题

**✅ 正面：**「做一个帮电商运营快速生成护肤品详情页逐屏文案的 Skill」—— 明确、具体

## 二、需求翻译：把口语化诉求翻译成 Skill 能力

用户不会说「我要一个 Skill」，他们只会描述痛点，你需要把痛点翻译成可交付的功能。

### 2.1 核心问题

- 用户的口语化表达背后，真实的需求是什么？
- 哪些需求是核心的，哪些是次要的？
- 怎么把模糊的诉求拆解成具体的功能模块？

### 2.2 翻译方法

1. **倾听原话**：记录用户的原始表达，不要先入为主
2. **提炼痛点**：找出用户反复提到的「麻烦」「费时间」「不统一」
3. **翻译成功能**：每个痛点对应一个 Skill 能力点
4. **排序优先级**：区分「必须有」和「nice to have」

### 2.3 检查清单

- [ ] 把用户的每一个痛点都翻译成了对应的功能
- [ ] 区分了核心功能和次要功能
- [ ] 功能描述是可验证的，不是模糊的
- [ ] 没有过度设计，只做用户真正需要的

### 2.4 示例

| 用户原话 | 翻译成 Skill 功能 |
|----------|-------------------|
| 「每次排版都要手调，太耗时间」 | 一键生成标准化排版 |
| 「不同人做出来的风格不统一」 | 内置品牌规范和设计令牌 |
| 「复制过去格式就乱了」 | 输出可直接粘贴的格式化内容 |

## 三、价值论证：为什么这个 Skill 值得做

不是所有需求都值得做成 Skill，需要判断它的复用价值和解决的痛点程度。

### 3.1 核心问题

- 这个 Skill 能帮用户省多少时间/多少钱？
- 使用频率有多高？
- 目标用户群体有多大？
- 相比现有方案，优势有多大？

### 3.2 价值判断三维度

| 高频 | 痛点深 | 可标准化 |
|------|--------|----------|
| 用户会反复使用，不是一次性需求 | 当前方案很麻烦，用户愿意为解决它付费 | 流程相对固定，能被工具化封装 |

### 3.3 检查清单

- [ ] 确认这是一个高频重复的需求
- [ ] 用户当前的解决方案有明显痛点
- [ ] 这个流程可以被标准化、工具化
- [ ] 目标用户群体足够大（或对特定用户价值足够高）

## 四、成品定义：Skill 应该长什么样

明确 Skill 的输入、输出、模块划分、参数设计，让用户一看就知道怎么用。

### 4.1 核心问题

- 用户需要输入什么？最少输入是什么？
- Skill 会输出什么？格式是什么？
- Skill 内部有哪些功能模块？
- 用户的使用流程是怎样的？

### 4.2 成品结构

#### 输入区

- 必填参数：用户必须提供的信息
- 可选参数：用户可以自定义的选项
- 预设模板：常见场景的快速选择

#### 处理逻辑

- 核心算法/规则：Skill 怎么处理输入
- 内置知识库：Skill 自带的参考数据
- 异常处理：输入不对时怎么办

#### 输出区

- 主输出：用户最想要的结果
- 辅助输出：额外的有用信息
- 导出格式：支持哪些格式的导出

### 4.3 检查清单

- [ ] 输入项足够清晰，用户知道该填什么
- [ ] 输出格式符合用户的使用场景
- [ ] 模块划分合理，没有冗余
- [ ] 用户流程顺畅，不需要学习成本

## 五、初始描述：给 AI 的第一段提示词怎么写

怎么描述这个 Skill，让 AI 能准确理解并生成符合预期的实现。

### 5.1 核心问题

- 怎么让 AI 一下子明白这个 Skill 的定位？
- 哪些信息是必须在第一段就说清楚的？
- 怎么避免 AI 理解偏差？

### 5.2 提示词结构

1. **角色定位**：这个 Skill 是干什么的，给谁用的
2. **使用场景**：在什么情况下会用到这个 Skill
3. **核心痛点**：用户现在遇到了什么问题
4. **功能要求**：具体要实现哪些功能
5. **输出要求**：产出什么格式的结果
6. **风格要求**：整体的调性、审美、专业度
7. **约束条件**：不要做什么，有哪些边界

### 5.3 检查清单

- [ ] 第一段就能让 AI 明白 Skill 的核心定位
- [ ] 功能要求具体、可验证
- [ ] 输出格式明确
- [ ] 说了「不要做什么」来避免跑偏

## 六、验收标准：生成后怎么检查

Skill 做完了，怎么判断它是不是真的能用、好用。

### 6.1 核心问题

- 怎么验证 Skill 确实解决了用户的问题？
- 哪些地方最容易出问题？
- 用户体验够不够好？

### 6.2 验收维度

#### 功能验收

- 核心功能是否都实现了？
- 输入边界情况处理得怎么样？
- 输出结果是否符合预期？

#### 体验验收

- 用户能不能一眼看懂怎么用？
- 操作流程顺不顺畅？
- 有没有明显的下一步指引？

#### 质量验收

- 结果质量够不够专业？
- 有没有明显的错误或漏洞？
- 风格是否统一？

### 6.3 检查清单

- [ ] 跑通了一次完整的使用流程
- [ ] 测试了至少 3 种不同的输入场景
- [ ] 输出结果可以直接使用，不需要二次修改
- [ ] 用户不需要看说明书就能上手

## 七、迭代优化：怎么从 v1 升级到 v2

第一版跑通之后，往哪些方向打磨能让 Skill 更专业、更值钱。

### 7.1 核心问题

- v1 解决了核心问题，v2 该优化什么？
- 哪些优化能带来最大的价值提升？
- 怎么避免越做越复杂？

### 7.2 优化方向

| 深度优化 | 体验优化 |
|----------|----------|
| 增加更多预设模板 | 简化操作流程 |
| 优化输出质量 | 增加示例内容 |
| 支持更多输入格式 | 优化视觉呈现 |
| 增加智能推荐 | 增加使用提示 |

### 7.3 优化原则

> 💡 **核心原则：** 始终围绕「让用户更快、更省心地完成任务」来优化，而不是堆功能。

- 每增加一个功能，都要问：用户真的需要吗？
- 优先优化高频使用的路径
- 保持简单，不要做成复杂系统

### 7.4 检查清单

- [ ] 优化方向是用户真实需要的，不是自嗨
- [ ] 没有过度设计，保持 Skill 的轻量性
- [ ] 优化后用户能明显感受到价值提升

---

## 八、交付物定义：最终产出什么

一个完整的 Skill 交付包应该包含哪些文件和文档。

### 8.1 核心交付物

| 交付物 | 说明 | 必要性 |
|--------|------|--------|
| SKILL.md | Skill 的说明文档，包含功能介绍、使用方法、参数说明 | 必须 |
| 主脚本/入口文件 | Skill 的核心实现代码 | 必须 |
| 示例/演示 | 让用户快速看到效果的示例内容 | 推荐 |
| 使用说明 | 简短的上手指南 | 推荐 |
| 素材资源 | Skill 用到的图片、模板、配置文件等 | 按需 |

### 8.2 交付标准

- 用户拿到就能用，不需要额外配置
- 有示例，用户能快速理解价值
- 文档清晰，遇到问题能自查

### 8.3 检查清单

- [ ] 核心文件齐全，能独立运行
- [ ] 有示例内容，用户能快速体验
- [ ] 文档完整，说明清楚
- [ ] 交付包结构清晰，容易理解

## 九、使用场景：Skill 适合在什么情况下用

明确 Skill 的适用边界，什么时候该用、什么时候不该用。

### 9.1 适用场景

- 高频重复的任务：每次都要做，且流程相对固定
- 标准化输出：需要统一风格、格式、质量的内容
- 效率提升：能明显减少人工操作时间
- 降低门槛：让非专业人士也能做出专业结果

### 9.2 不适用场景

- 一次性需求：做完就完了，不值得封装
- 高度创造性的工作：需要每次都全新创作，没有固定流程
- 太复杂的系统：应该用完整的应用，而不是 Skill
- 需要实时数据交互的：Skill 适合离线处理

### 9.3 检查清单

- [ ] 明确说了这个 Skill 适合什么场景
- [ ] 明确说了不适合什么场景
- [ ] 用户能快速判断自己的需求是否匹配

---

## 十、发布与推广：怎么让用户知道并使用这个 Skill

做好的 Skill 怎么交付给用户，怎么让用户感受到价值。

### 10.1 交付方式

1. **演示先行**：先给用户看效果，再讲功能
2. **一句话价值**：用一句话说清楚这个 Skill 能帮他省多少事
3. **示例体验**：给一个现成的示例，让用户直接感受
4. **使用引导**：告诉用户第一步该做什么

### 10.2 价值话术

**❌ 反面：**「我给你做了一个 Skill，功能很强大，有很多模块」

**✅ 正面：**「这个 Skill 能帮你把每次 2 小时的排版工作缩到 5 分钟，而且风格完全统一」

### 10.3 检查清单

- [ ] 有清晰的价值主张，用户一眼能看到好处
- [ ] 有示例或演示，用户能快速体验
- [ ] 使用门槛低，用户上手就能用
- [ ] 有明确的下一步行动指引

## 附录：完整 SOP 检查清单

设计一个 Skill 时，按顺序检查以下所有条目：

### 定位阶段

- [ ] 能用一句话说清楚 Skill 的定位
- [ ] 明确了目标用户和核心痛点

### 需求阶段

- [ ] 把用户痛点翻译成了具体功能
- [ ] 区分了核心功能和次要功能

### 价值阶段

- [ ] 确认这是高频、痛点深、可标准化的需求
- [ ] 相比现有方案有明显优势

### 设计阶段

- [ ] 输入输出清晰明确
- [ ] 模块划分合理
- [ ] 用户流程顺畅

### 实现阶段

- [ ] 提示词描述准确完整
- [ ] 核心功能都实现了

### 验收阶段

- [ ] 跑通了完整使用流程
- [ ] 测试了多种输入场景
- [ ] 输出质量达标

### 交付阶段

- [ ] 交付物齐全
- [ ] 有示例和使用说明
- [ ] 价值主张清晰
