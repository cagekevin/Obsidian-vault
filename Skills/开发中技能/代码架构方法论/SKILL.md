# 代码架构方法论（S5 精华提炼）

来源于 S5-代码架构优化 的核心思想体系。

## 核心术语体系（唯一真相）

| 术语 | 定义 | 别说成 |
|------|------|-------|
| **Module（模块）** | 任何有接口和实现的东西：函数、类、包、跨层切片 | component, service, unit |
| **Interface（接口）** | 调用者需要知道的全部：类型签名 + 不变量 + 顺序约束 + 错误模式 + 配置 | API, signature |
| **Implementation（实现）** | 模块内部的代码体 | — |
| **Depth（深度）** | 接口层面的杠杆率——小接口背后的大行为。**Deep = 高杠杆** | — |
| **Shallow（浅层）** | 接口几乎和实现一样复杂（反面深度） | — |
| **Seam（接缝）** | 接口所在的位置；在不修改原地的情况下改变行为的地点 | boundary |
| **Adapter（适配器）** | 在接缝处满足接口的具体实现 | — |
| **Leverage（杠杆）** | 调用者从深度中获得的好处 | — |
| **Locality（局域性）** | 维护者从深度中获得的好处：变更、bug、知识集中在一处 | — |

## 三大核心原则

### 1. 删除测试（Deletion Test）
> 想象删除这个模块。如果复杂性消失了，说明它只是个透传。如果复杂性重新分散到 N 个调用方，说明它在创造价值。

用来判断模块是否"浅"：如果把这层去掉，逻辑就散到外面去了 → 它在发挥作用。

### 2. 接口即测试面（The Interface Is the Test Surface）
> 调用者和测试跨越同一个接缝。如果你想"绕过"接口去测试，模块形状大概率有问题。

测试应该对着接口写，而不是对着内部实现。如果测试必须跟着内部实现改，说明测试越过了接口。

### 3. 一/二适配器规则（One/Two Adapter Rule）
> **一个适配器 = 假设性接缝。两个适配器 = 真正的接缝。**

不要为一个仅仅在理论上需要的抽象引入接缝（YAGNI）。只有当至少有两个具体实现（通常是 production + test）时，接缝才值得存在。

## 模块依赖分类（决定接缝策略）

| 类型 | 说明 | 接缝策略 |
|------|------|---------|
| **In-process** | 纯计算、内存状态、无 I/O | 总能深化——直接合并模块，通过新接口测试。不需要适配器 |
| **Local-substitutable** | 有本地测试替身的依赖（PGLite、内存文件系统） | 如果替身存在就可以深化。接缝在内部，不暴露到外部接口 |
| **Remote but owned（Ports & Adapters）** | 自己的跨网络服务（微服务、内部 API） | 定义端口（接口），传输层注入为适配器。测试用内存适配器，生产用 HTTP/gRPC |
| **True external（Mock）** | 不受你控制的第三方服务（Stripe、Twilio） | 将外部依赖注入为端口；测试提供 mock 适配器 |

## 深化方法论（如何把浅变深）

### 判断信号：什么是"浅"？
- 理解一个概念需要在多个小模块之间跳转
- 接口几乎和实现一样复杂
- 纯函数被抽出来仅仅为了可测试性，但真正的 bug 藏在调用方式里（缺乏局域性）
- 紧密耦合的模块在跨接缝泄漏
- 代码难以测试，或者只能通过复杂接口测试

### 深化策略
1. 把多个浅模块合并成一个深模块
2. 新模块暴露小接口（高杠杆）
3. 旧单元测试变成冗余 → 删掉，在新接口层面重写
4. 测试应该能扛住内部重构——测试描述的是行为，不是实现

### 接口设计：设计两次
第一次想到的方案通常不是最好的。并行探索多个不同方向的接口设计：

| 方向 | 约束 |
|------|------|
| 接口最小化 | 1-3 个入口点，最大化每个入口点的杠杆 |
| 灵活性最大化 | 支持多种用例和扩展 |
| 优化最常见的调用方 | 让默认路径变得极其简单 |
| Ports & Adapters | 针对跨接缝依赖设计 |

## 报告输出

架构审查输出为自包含 HTML 报告（存于 `Temp/`），每项候选包含：
- **Files** — 涉及的文件/模块
- **Problem** — 当前架构的痛点
- **Solution** — 白描方案
- **Benefits** — 用局域性和杠杆率解释收益
- **Before / After 对比图**
- **推荐强度** — Strong / Worth exploring / Speculative
