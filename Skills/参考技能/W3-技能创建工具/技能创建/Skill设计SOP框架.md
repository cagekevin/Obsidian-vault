# Skill 设计 SOP 框架

> 基于产品化思维的 Skill 设计标准作业流程，从定位到交付的完整方法论。

---

## 一、定位阶段：明确 Skill 要解决什么问题

这是设计 Skill 的第一步，核心是搞清楚「这个 Skill 到底解决谁的什么问题」。

### 1.1 核心问题

- 这个 Skill 解决的是哪类人的什么具体问题？
- 是一次性需求还是高频重复需求？
- 用户现在是怎么解决这个问题的？痛点有多痛？

### 1.2 输入与输出

| 类型 | 内容 |
|------|------|
| 输入 | 用户的模糊需求、痛点描述、使用场景 |
| 输出 | 一句话定位：「这是一个帮 [谁] 解决 [什么问题] 的 Skill」 |

### 1.3 检查清单

- [ ] 能用一句话说清楚这个 Skill 是干什么的
- [ ] 明确了目标用户是谁
- [ ] 找到了用户当前的替代方案和痛点
- [ ] 确认这是一个可被工具化的重复流程

### 1.4 正反示例

**❌ 反面：**「做一个设计 Skill」—— 太宽泛，不知道解决什么问题

**✅ 正面：**「做一个帮电商运营快速生成护肤品详情页逐屏文案的 Skill」—— 明确、具体

## 二、需求翻译：把口语化诉求翻译成 Skill 能力

用户不会说「我要一个 Skill」，他们只会描述痛点，你需要把痛点翻译成可交付的功能。

### 2.1 核心问题

- 用户的口语化表达背后，真实的需求是什么？
- 哪些需求是核心的，哪些是次要的？
- 怎么把模糊的诉求拆解成具体的功能模块？

### 2.2 翻译方法

1. **倾听原话**：记录用户的原始表达，不要先入为主
2. **提炼痛点**：找出用户反复提到的「麻烦」「费时间」「不统一」
3. **翻译成功能**：每个痛点对应一个 Skill 能力点
4. **排序优先级**：区分「必须有」和「nice to have」

### 2.3 检查清单

- [ ] 把用户的每一个痛点都翻译成了对应的功能
- [ ] 区分了核心功能和次要功能
- [ ] 功能描述是可验证的，不是模糊的
- [ ] 没有过度设计，只做用户真正需要的

### 2.4 示例

| 用户原话 | 翻译成 Skill 功能 |
|----------|-------------------|
| 「每次排版都要手调，太耗时间」 | 一键生成标准化排版 |
| 「不同人做出来的风格不统一」 | 内置品牌规范和设计令牌 |
| 「复制过去格式就乱了」 | 输出可直接粘贴的格式化内容 |

## 三、价值论证：为什么这个 Skill 值得做

不是所有需求都值得做成 Skill，需要判断它的复用价值和解决的痛点程度。

### 3.1 核心问题

- 这个 Skill 能帮用户省多少时间/多少钱？
- 使用频率有多高？
- 目标用户群体有多大？
- 相比现有方案，优势有多大？

### 3.2 价值判断三维度

| 高频 | 痛点深 | 可标准化 |
|------|--------|----------|
| 用户会反复使用，不是一次性需求 | 当前方案很麻烦，用户愿意为解决它付费 | 流程相对固定，能被工具化封装 |

### 3.3 检查清单

- [ ] 确认这是一个高频重复的需求
- [ ] 用户当前的解决方案有明显痛点
- [ ] 这个流程可以被标准化、工具化
- [ ] 目标用户群体足够大（或对特定用户价值足够高）

## 四、成品定义：Skill 应该长什么样

明确 Skill 的输入、输出、模块划分、参数设计，让用户一看就知道怎么用。

### 4.1 核心问题

- 用户需要输入什么？最少输入是什么？
- Skill 会输出什么？格式是什么？
- Skill 内部有哪些功能模块？
- 用户的使用流程是怎样的？

### 4.2 成品结构

#### 输入区

- 必填参数：用户必须提供的信息
- 可选参数：用户可以自定义的选项
- 预设模板：常见场景的快速选择

#### 处理逻辑

- 核心算法/规则：Skill 怎么处理输入
- 内置知识库：Skill 自带的参考数据
- 异常处理：输入不对时怎么办

#### 输出区

- 主输出：用户最想要的结果
- 辅助输出：额外的有用信息
- 导出格式：支持哪些格式的导出

### 4.3 检查清单

- [ ] 输入项足够清晰，用户知道该填什么
- [ ] 输出格式符合用户的使用场景
- [ ] 模块划分合理，没有冗余
- [ ] 用户流程顺畅，不需要学习成本

## 五、工作流设计：Skill 内部怎么流转

明确 Skill 拿到用户输入后，内部是怎么判断、怎么流转、怎么应对各种情况的。这是 Skill 体验的核心。

### 5.1 核心问题

- 用户可能会有哪些意图？
- 每个意图对应什么处理流程？
- 信息不够时怎么问？
- 用户否定/反馈时怎么应对？
- 出问题时怎么兜底？

### 5.2 意图清单

先把所有可能的用户意图列出来，确保互斥、不重叠。

| 意图类型 | 说明 | 示例 |
|----------|------|------|
| 新建/生成 | 用户要从零开始生成内容 | "给我写一个场景"、"出一张主图" |
| 修改/编辑 | 用户对已有结果做调整 | "改一下这句台词"、"把颜色换成金色" |
| 否定/不满意 | 用户说不行、不对、不好 | "不对"、"不是这个感觉"、"重来" |
| 删减/压缩 | 用户要缩短、精简 | "短一点"、"把这个场景剪掉" |
| 提问/想法 | 用户提了一个新想法或问题 | "我有个想法..."、"如果这样行不行？" |
| 审计/检查 | 用户要做质量检查 | "帮我看看因果链对不对"、"检查一下有没有问题" |
| 信息不足 | 用户提供的信息不够，需要补全 | （系统判断，不是用户主动说的） |

### 5.3 主流程设计

每个意图对应一套完整的处理流程。

**设计要点：**
1. **第一步做什么** — 是先读上下文？还是先检查参数？
2. **核心处理步骤** — 分几步完成，每一步的输入输出是什么
3. **输出什么** — 主输出 + 辅助输出（分析、说明、建议）
4. **最后一步做什么** — 是问用户下一步？还是直接结束？

**示例："新建"意图的流程**
```
1. 读取上下文（已有资料、历史记录）
2. 检查关键参数是否齐全
3. 执行核心生成逻辑
4. 输出生成结果 + 简要分析
5. 问用户具体要改哪里（窄问题，不是开放式）
```

### 5.4 异常与反馈流程

正常流程之外的情况，也要设计清楚。

#### 信息不足时
- 不要瞎编，不要硬猜
- 问**一个**窄的二元问题，不是开放式问题
- 问完就等，不要同时问多个

❌ "你想要什么风格？"（太开放）
✅ "你想要奢华风还是简约风？"（二元，窄）

#### 用户否定时
- 不要过度道歉，一句"明白"就够了
- 不要一下甩出 5 个版本让用户选
- 问**一个**问题定位问题：是方向？是节奏？是具体某一处？
- 得到答案后，给一个方案，不是五个

#### 边界情况
- 输入格式不对怎么办？
- 内容超出范围怎么办？
- 用户要求的功能 Skill 做不了怎么办？

### 5.5 迭代模式

用户反复修改时，怎么迭代？

**两种模式：**
- **点式修改**：只改用户说的那一处，其他不动。适用于微调、改具体内容
- **全量重写**：整个重新生成。适用于方向大改、第一版完全不对

**设计原则：**
- 默认点式修改，不要顺便优化用户没让改的地方
- 全量重写要谨慎，说明为什么要重写
- 每次迭代都要有明确的版本感

### 5.6 防错机制

把常见的坑在设计阶段就堵死。

**常见的防错手段：**
- **钢铁红线**：绝对不能做的事，写在最前面
- **排他门禁**：某些形态的产品不能用某些模板，直接拦截
- **容量熔断**：输入超过一定数量，自动降级或降噪
- **自检清单**：输出前自己检查一遍，常见错误自动修正

### 5.7 检查清单

- [ ] 所有主要用户意图都列出来了（至少覆盖 80% 的使用场景）
- [ ] 每个意图都有清晰的处理流程，第一步和最后一步明确
- [ ] 信息不足时有明确的提问策略（问什么、怎么问）
- [ ] 用户否定时有应对流程（不道歉、不甩 5 个版本、问一个问题）
- [ ] 迭代模式明确（默认点式修改）
- [ ] 至少有 3 条防错机制
- [ ] 边界情况有兜底方案

## 六、初始描述：给 AI 的第一段提示词怎么写

怎么描述这个 Skill，让 AI 能准确理解并生成符合预期的实现。

### 6.1 核心问题

- 怎么让 AI 一下子明白这个 Skill 的定位？
- 哪些信息是必须在第一段就说清楚的？
- 怎么避免 AI 理解偏差？

### 6.2 提示词结构

1. **角色定位**：这个 Skill 是干什么的，给谁用的
2. **使用场景**：在什么情况下会用到这个 Skill
3. **核心痛点**：用户现在遇到了什么问题
4. **功能要求**：具体要实现哪些功能
5. **输出要求**：产出什么格式的结果
6. **风格要求**：整体的调性、审美、专业度
7. **约束条件**：不要做什么，有哪些边界

### 6.3 检查清单

- [ ] 第一段就能让 AI 明白 Skill 的核心定位
- [ ] 功能要求具体、可验证
- [ ] 输出格式明确
- [ ] 说了「不要做什么」来避免跑偏

---

## 七、验收标准：生成后怎么检查

Skill 做完了，怎么判断它是不是真的能用、好用。

### 7.1 核心问题

- 怎么验证 Skill 确实解决了用户的问题？
- 哪些地方最容易出问题？
- 用户体验够不够好？

### 7.2 验收维度

#### 功能验收

- 核心功能是否都实现了？
- 输入边界情况处理得怎么样？
- 输出结果是否符合预期？

#### 体验验收

- 用户能不能一眼看懂怎么用？
- 操作流程顺不顺畅？
- 有没有明显的下一步指引？

#### 质量验收

- 结果质量够不够专业？
- 有没有明显的错误或漏洞？
- 风格是否统一？

### 7.3 检查清单

- [ ] 跑通了一次完整的使用流程
- [ ] 测试了至少 3 种不同的输入场景
- [ ] 输出结果可以直接使用，不需要二次修改
- [ ] 用户不需要看说明书就能上手

---

## 八、迭代优化：怎么从 v1 升级到 v2

第一版跑通之后，往哪些方向打磨能让 Skill 更专业、更值钱。

### 8.1 核心问题

- v1 解决了核心问题，v2 该优化什么？
- 哪些优化能带来最大的价值提升？
- 怎么避免越做越复杂？

### 8.2 优化方向

| 深度优化 | 体验优化 |
|----------|----------|
| 增加更多预设模板 | 简化操作流程 |
| 优化输出质量 | 增加示例内容 |
| 支持更多输入格式 | 优化视觉呈现 |
| 增加智能推荐 | 增加使用提示 |

### 8.3 优化原则

> 💡 **核心原则：** 始终围绕「让用户更快、更省心地完成任务」来优化，而不是堆功能。

- 每增加一个功能，都要问：用户真的需要吗？
- 优先优化高频使用的路径
- 保持简单，不要做成复杂系统

### 8.4 检查清单

- [ ] 优化方向是用户真实需要的，不是自嗨
- [ ] 没有过度设计，保持 Skill 的轻量性
- [ ] 优化后用户能明显感受到价值提升

---

## 九、交付物定义：最终产出什么

一个完整的 Skill 交付包应该包含哪些文件和文档。

### 9.1 核心交付物

| 交付物 | 说明 | 必要性 |
|--------|------|--------|
| SKILL.md | Skill 的说明文档，包含功能介绍、使用方法、参数说明 | 必须 |
| 主脚本/入口文件 | Skill 的核心实现代码 | 必须 |
| 示例/演示 | 让用户快速看到效果的示例内容 | 推荐 |
| 使用说明 | 简短的上手指南 | 推荐 |
| 素材资源 | Skill 用到的图片、模板、配置文件等 | 按需 |

### 9.2 交付标准

- 用户拿到就能用，不需要额外配置
- 有示例，用户能快速理解价值
- 文档清晰，遇到问题能自查

### 9.3 检查清单

- [ ] 核心文件齐全，能独立运行
- [ ] 有示例内容，用户能快速体验
- [ ] 文档完整，说明清楚
- [ ] 交付包结构清晰，容易理解

## 十、使用场景：Skill 适合在什么情况下用

明确 Skill 的适用边界，什么时候该用、什么时候不该用。

### 10.1 适用场景

- 高频重复的任务：每次都要做，且流程相对固定
- 标准化输出：需要统一风格、格式、质量的内容
- 效率提升：能明显减少人工操作时间
- 降低门槛：让非专业人士也能做出专业结果

### 10.2 不适用场景

- 一次性需求：做完就完了，不值得封装
- 高度创造性的工作：需要每次都全新创作，没有固定流程
- 太复杂的系统：应该用完整的应用，而不是 Skill
- 需要实时数据交互的：Skill 适合离线处理

### 10.3 检查清单

- [ ] 明确说了这个 Skill 适合什么场景
- [ ] 明确说了不适合什么场景
- [ ] 用户能快速判断自己的需求是否匹配

---

## 十一、发布与推广：怎么让用户知道并使用这个 Skill

做好的 Skill 怎么交付给用户，怎么让用户感受到价值。

### 11.1 交付方式

1. **演示先行**：先给用户看效果，再讲功能
2. **一句话价值**：用一句话说清楚这个 Skill 能帮他省多少事
3. **示例体验**：给一个现成的示例，让用户直接感受
4. **使用引导**：告诉用户第一步该做什么

### 11.2 价值话术

**❌ 反面：**「我给你做了一个 Skill，功能很强大，有很多模块」

**✅ 正面：**「这个 Skill 能帮你把每次 2 小时的排版工作缩到 5 分钟，而且风格完全统一」

### 11.3 检查清单

- [ ] 有清晰的价值主张，用户一眼能看到好处
- [ ] 有示例或演示，用户能快速体验
- [ ] 使用门槛低，用户上手就能用
- [ ] 有明确的下一步行动指引
