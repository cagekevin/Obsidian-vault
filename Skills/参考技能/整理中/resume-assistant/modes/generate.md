# mode: generate · 从 0 生成一份新简历

> **前置**：先读 [`_shared.md`](_shared.md)（15 条原则 + 通用工作流骨架 + NEVER 清单）。本文件只讲 generate 的差异。

## 触发条件

- 用户说："帮我写简历 / 生成一份简历"（未指定 JD）
- `_manifest.json` 尚不存在、或其 `master_version_id` 为空
- 用户明确要求 `mode: "generate"`（显式覆盖）

## 输出定位

**首次 generate → 自动写入 `resume-output/_master/`**，`is_master: true`。
这是所有后续 tailor / rewrite 的 lineage root。

---

## 关键次序：信息收集 → Preflight → 生成

> ⚠️ **generate 模式的特殊流程要求**：Preflight 三选项（模板/长度/语言）依赖用户的经历密度做 fallback 启发式推断，因此**必须先完成用户信息收集，再发出 Preflight 问询**，顺序不可颠倒。
>
> 正确次序：
> 1. **Step 0.1 信息收集**（本文件 §信息收集脚本）
> 2. **Step 0.5 Preflight**（`references/preflight-questions.md`）
> 3. **Step 1-8 正式生成流程**

---

## 信息收集脚本（Step 0.1 · generate 专属）

触发 generate 后，**第一件事**是收集用户的原始素材。一次性发出以下问询，**不要逐项追问**：

```
好的，我来帮你生成一份简历！请提供以下信息（能提供多少就提供多少，不确定的直接跳过）：

【基础信息】
- 姓名：
- 电话：（如不提供，将用占位符 [待填写]）
- 邮箱：（如不提供，将用占位符 [待填写]）
- 所在城市：
- GitHub / 个人网站：（可选）

【教育背景】（至少 1 条）
- 学校、专业、学历、在读时间（如 2020.09 – 2024.06）：
- GPA / 排名 / 获奖：（可选）

【工作 / 项目经历】（至少 1 条，可多条）
- 公司 / 项目名、职位 / 角色、时间：
- 主要做了什么？（口语描述即可，AI 会帮你润色）
- 有没有量化数据？（用了多少资源、提升了多少 %、服务了多少用户等）
- 用到了哪些技术栈 / 工具？

【专业技能】（可选）
- 编程语言、框架、数据库、云平台等：

如果你有现成简历文本，直接粘贴也行，我来帮你解析整理。
```

### 信息收集后的检查清单

收到用户回复后，检查以下项目，**缺项则追问一次**（只追问一次，不反复催）：

| 字段 | 缺失处理 |
|---|---|
| 姓名 | 追问一次；仍未提供 → 用 `[姓名]` 占位 |
| 电话 | **不追问**；直接用 `[电话待填写]` 占位，并在 Step 8 提醒用户填写 |
| 邮箱 | **不追问**；直接用 `[邮箱待填写]` 占位，并在 Step 8 提醒用户填写 |
| 教育背景 | 缺学历/专业/时间 → 追问一次；仍缺 → 用 `[待补充]` 占位 |
| 工作/项目经历 | 至少 1 条，否则无法继续 → 必须追问 |
| 量化数据 | 有经历但无数字 → 追问一次（"这段经历有没有量化数据，比如项目规模、提升了多少%？"）；用户说没有 → 用 `____` 占位符，**绝不编造** |

## 个人信息占位符规则（防幻觉 · generate 最高优先级）

> **核心原则**：AI **绝不编造**姓名、电话、邮箱、微信、地址等个人联系信息。
> 这类信息一旦编造，用户提交简历后会被 HR 直接识破，后果严重。

| 字段 | 用户未提供时的处理 |
|---|---|
| 姓名 | `[姓名]` |
| 电话 | `[电话待填写]` |
| 邮箱 | `[邮箱待填写]` |
| 微信 | 直接省略该行 |
| GitHub | 直接省略该行 |
| LinkedIn | 直接省略该行 |
| 个人网站 | 直接省略该行 |
| 城市 | `[城市]`（会影响 ATS 地域过滤，建议用户填写）|
| 期望薪资 | 直接省略（不占位，不编造）|
| GPA / 排名 | `[待补充]`（不推测、不美化）|

**生成后在 Step 8 审核环节，必须高亮提醒用户**：

```
⚠️ 以下占位符需要你自行填写真实信息：
- [电话待填写] → 你的手机号
- [邮箱待填写] → 你的常用邮箱
- [其他占位符] → ...
请在交付给 HR 前务必替换。
```

## 与骨架的差异（逐步标注）

| Step | 差异 |
|---|---|
| **Step 0.1** | **信息收集**（本文件 §信息收集脚本）**必须先于 Preflight 完成** |
| Step 0 | mode 已定，跳过 mode 确认 |
| Step 0.5 | Preflight 三选项（基于已收集的经历信息做 fallback 启发式） |
| Step 1 | **严格校验**：姓名 + education + ≥1 work/project 缺一不可；联系方式缺失用占位符（见上表） |
| **Step 2** | **跳过 JD 解析**（generate 不绑 JD）|
| **Step 3** | 只能依赖用户 experiences 里的 self-declared role_family（如未提供 → 根据最近一段经历推断） |
| Step 4 | 模板按 Preflight 选项；未指定 → `preferences.template || "hybrid"`；`language` 按 `preferences.language || "zh"` |
| Step 5 | 三维度 Provenance **全部必跑**（首版最容易幻觉） |
| **Step 6** | **跳过 JD 覆盖率验证** |
| Step 7 | 写入 `resume-output/_master/`；`is_master: true`；`parent_id: null`；不生成 `diff-report.html`（无对比对象）|
| Step 8 | 审核要点额外强调："这是你的主档简历，后续所有定制版都从它派生，请特别确认事实准确"；**必须高亮列出所有占位符，提示用户手动填写** |

## 典型对话开场

> 用户："帮我写一份简历。"
>
> Skill（应）：
> 1. 确认这是首份简历（无 `_master/`）→ mode = generate
> 2. **发出信息收集脚本**（一次性收齐，不逐项催）
> 3. 收到用户填写后，检查缺口（量化数据 / 技术栈），追问一次
> 4. **发出 Preflight 三选项**（此时已有经历信息，fallback 启发式可正常运行）
> 5. 用 Preflight 结果写出 master；零 JD → `coverage_before` 字段不生成
> 6. 进入 Step 5 三维度审计 → Step 8 让用户逐条确认 + **高亮所有占位符**

## 只在 generate 独有的 NEVER

- **NEVER** 在没有收集用户信息的情况下直接触发 Preflight（信息收集必须先于 Preflight）
- **NEVER** 编造用户未提供的联系信息（电话/邮箱/微信/地址）——统一用占位符
- **NEVER** 因为 `email` 在 schema 中标注为"必填"就自己生成一个假邮箱——占位符 `[邮箱待填写]` 是合法状态
- **NEVER** 在 generate 时写入 `resume-output/<version_label>/`（那是派生版目录）
- **NEVER** 在 generate 时生成 `jd-match-report.md`（无 JD 来源）
- **NEVER** 在 generate 时让 `is_master = false`（除非用户显式指示且 `_manifest` 已有其他 master）

## 成功判定

- `_manifest.json.master_version_id` 指向本次生成的 `version_id`
- `processing_status = ready`
- 用户在 Step 8 全部 `approved`
- 三维度审计均过（第三维可有 INFO 残留，不影响）
- Step 8 已高亮列出所有占位符并告知用户填写
