# lessons-learned · plan-tracker 真实踩坑案例

> 这里收录的是 **真实迭代中遇到的问题** 和最终修正方案，不是教科书理论。
> 陪伴类 skill 的失败比规划类更隐蔽——用户不会告诉你哪里不爽，他们直接卸载/删除。

---

## Case 1：「严格教练」人设的一句话让用户当晚卸载

### 现象（v0.2 真实事件）

- 一位用户配置 persona = `strict-coach`，准备考研冲刺
- 考研倒计时 60 天，用户连续打卡 12 天，状态良好
- 第 13 天用户加班到凌晨没打卡，第 14 天又出差没打卡
- 第 15 天用户回家打开 plan-tracker，触发了断签恢复流程

**v0.2 的 revive.py 输出（错误版）**：

```
教练：连续 12 天的努力被你 2 天破功，对得起距离考研只剩 45 天的自己吗？
明天必须把这两天落下的任务全部补回来，否则永远赶不上进度。
```

- 用户当晚截图发到群里："这玩意儿是不是有毛病？我加班这么累还要被骂？"
- **当晚卸载，再也没回来**
- 内部追溯：那 2 天她失去的不只是 streak，是 4 年坚持考研的最后一根稻草

### 根因

1. ❌ **第一次以为**：「严格教练就该严厉一点啊？」 → 错。严厉不等于羞辱。
2. ✅ **真因**：v0.2 没有区分 **「严格的方法论」** 和 **「严苛的人格」**。
   - 真实优秀的教练：「你这周训练量没达标，我们调整一下下周计划」
   - 假冒严格的输出：「你不配 / 你白练了 / 你对不起谁」← 这是 PUA，不是 strict-coach

### 修复（v0.3 起）

1. ✅ 重写 `references/personas/strict-coach.md`，明确划线：
   - **可以**：直接、专业、不打感情牌、用数据说话
   - **不可以**：贬低、羞辱、人身评价、施加情感重量
2. ✅ 所有 persona 文件加上 **共用底线**（非 persona-specific）：
   ```
   通用红线（任何人设都不可越界）：
   - 不指责用户的人格 / 价值
   - 不把 streak 当道德武器
   - 用户表达疲惫时，先共情后建议
   - 不主动鞭打"还差多少"的进度焦虑
   ```
3. ✅ 在 SKILL.md 上升为 **NEVER 1**「Streak 当威胁/羞辱用户」

### 防回归

- `evals/persona-evals.md`：每个 persona 加 "断签恢复场景" 测试用例
- 黑盒测试：让另一个 AI 扮演用户说「我加班 2 天没打卡」，检查 4 种 persona 的回复**全部不能**包含：贬义形容词 / 道德评价 / "对不起谁" 类词组

### 教训

> **persona 的"严格" ≠ "刻薄"，persona 的"幽默" ≠ "嘲讽"，persona 的"温柔" ≠ "迁就"**。
> 陪伴类 skill 的人设设计，第一步要画的不是"个性",而是**"红线"——什么话所有人设都不能说**。
> 一个生产级陪伴 skill 的失败，往往不是因为它"陪得不够好"，而是因为它**在错误的瞬间说错了一句话**。

## Case 2：周报「每周都说很棒」让 4 个用户连续 3 周没察觉自己在掉队

### 现象（v0.4 真实数据）

- 4 名用户备考雅思，使用 v0.4 的 weekly 周报功能
- 4 人本周完成率分别为：38% / 45% / 51% / 60%（**全部低于目标 70%**）
- v0.4 的周报模板（错误版）：

  ```
  本周亮点：
    - 你完成了 X 项任务！
    - 累计学习时长 N 分钟，棒棒哒～
    - 继续保持这份热情！
  ```

- 4 人都觉得「我这周还行」，下周继续以同样节奏学
- 第 3 周后追踪：**4 人有 3 人考试成绩低于预期**，事后访谈一致反馈：「我以为我学得挺好的，工具没告诉我有问题」

### 根因

- v0.4 的周报只挑「亮点」展示，避开「问题」（产品经理的本能：用户喜欢被夸）
- 但是用户开陪伴 skill 不只是为了被夸——**他在赌一份「真实的诊断」**
- 报喜不报忧 = AI 在帮用户**隐瞒坏消息**，等于把用户的命运赌在虚假繁荣上

### 修复（v0.5 - v1.0）

1. ✅ 周报模板硬性要求 **诚实呈现完成率百分比**（精确到个位数，不四舍五入到好看数字）
2. ✅ 当完成率 < 70% 时，必须包含三段：
   - 数据现状（不修饰）
   - 弱日识别（具体哪几天没打卡 + 推断原因）
   - 具体改进建议（不能只说"加油")
3. ✅ 用 persona 包装"难听的真话"——而不是用 persona 替换真话
   - 温柔学姐版：「这周完成 8/20，比目标低了一些。看下来周三周五都没打卡——是这两天工作日时段太紧吗？要不下周把预算调到 60 分钟？」
   - 严格教练版：「完成率 40%。诊断：周中两日缺勤。建议：调整 weekday daily_budget 至 60 分钟。」
   - 两者**数据完全一致**，只是语气不同
4. ✅ 在 SKILL.md 上升为 **NEVER 8**「周报报喜不报忧」

### 防回归

- `evals/stats-evals.md`：构造 4 个完成率不同的测试输入（30% / 50% / 70% / 90%），检查输出是否含真实百分比、是否含改进建议
- `tests/test_stats.py`：完成率 < 70% 时若输出不含 "建议" 关键词 → 测试失败

### 教训

> **陪伴 skill 的诚实度是信任基石**。用户对 AI 反馈的容忍度比对人类高得多——人类报喜不报忧叫情商，AI 报喜不报忧叫数据造假。
> 真实的"陪伴"不是哄着用户开心，而是**在用户即将走偏时，用他能接受的方式给他喊一声**。
> persona 是包装，不是滤镜——它可以让难听的真话变得**易于接受**，但不能让真话**消失**。

## 共性教训：为什么陪伴类 skill 比工具类更脆弱

| 工具类 skill 失败 | 陪伴类 skill 失败 |
|------------------|------------------|
| 用户报 bug，等修复 | 用户默默卸载，永不回头 |
| 失败可量化（崩溃 / 报错） | 失败不可见（情绪伤害） |
| 修复 = 改代码 | 修复 = 改人设 + 重建信任（很难） |

这就是为什么陪伴类 skill 的反模式套件（NEVER 列表）需要写得**比工具类更细、更狠、更克制**。
