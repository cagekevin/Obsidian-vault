# 逻辑原型

一个小型交互式终端应用，让用户手动驱动状态模型。当问题涉及**业务逻辑、状态转换或数据结构**时使用——这类问题在纸面上看起来合理，但只有通过实际案例推演后才会发现问题。

## 什么时候适合用

- "我不确定这个状态机是否能处理 X 然后 Y 的边界情况。"
- "这个数据模型真的能让我表示……的场景吗？"
- "我想在开始写代码之前先感觉一下 API 应该长什么样。"
- 任何用户想要**按按钮并观察状态变化**的场景。

如果问题是"应该长什么样"——走错分支了。请使用 [UI.md](UI.md)。

## 流程

### 1. 明确问题

在写代码之前，写下要原型化的状态模型和要回答的问题。一个段落，放在原型的 README 或文件顶部的注释中。回答了错误问题的逻辑原型就是纯粹的浪费——把问题明确写下来，以便后续可以检查，无论用户是在旁边观看还是稍后回来查看。

### 2. 选择语言

使用宿主项目使用的语言。如果项目没有明显的运行时（如文档仓库），询问。

与项目现有的工具链约定保持一致——不要仅仅为了原型而引入新的包管理器或运行时。

### 3. 将逻辑隔离在可移植模块中

将实际逻辑——回答问题的核心部分——放在一个小型、纯净的接口后面，以便日后可以提取出来放入真实代码库。围绕它的 TUI 是可丢弃的；逻辑模块则不应该被丢弃。

合适的形状取决于问题：

- **纯 reducer**——`(state, action) => state`。适用于动作为离散事件且状态为单一值的场景。
- **状态机**——明确的状态和转换。适用于"当前哪些操作是合法的"本身就是问题一部分的场景。
- **一组纯函数**，操作纯数据类型。适用于没有隐含当前状态的场景——只有转换。
- **拥有清晰方法表面的类或模块**，当逻辑确实拥有持续的内部状态时。

选择最符合问题的形状，*而不是*最容易接入 TUI 的那一个。保持纯净：无 I/O，无终端代码，无 `console.log`（用于控制流）。TUI 导入它并调用它；没有任何东西反向流入。

这正是让原型在其生命周期之外仍有价值的原因。当问题得到回答后，验证过的 reducer/状态机/函数集可以被提取到真实模块中——TUI 外壳则被删除。

### 4. 构建最小的能展示状态的 TUI

构建一个**轻量级 TUI**——每次 tick 时清屏（`console.clear()` / `print("\033[2J\033[H")` / 等价方式）并重新渲染整个画面。用户应始终看到一个稳定的视图，而不是不断增长的滚动缓冲区。

每帧包含两部分，按此顺序：

1. **当前状态**，经过美化打印且易于 diff（每行一个字段，或格式化 JSON）。**加粗**显示字段名或章节标题，**暗淡**显示不太重要的上下文（时间戳、ID、衍生值）。原生 ANSI 转义码即可——`\x1b[1m` 加粗、`\x1b[2m` 暗淡、`\x1b[0m` 重置。除非项目中已有样式库，否则无需引入。
2. **键盘快捷键**，显示在底部：`[a] 添加用户 [d] 删除用户 [t] 推进时间 [q] 退出`。按键加粗、说明暗淡，反之亦然——只要阅读清晰即可。

行为：

1. **初始化状态**——一个内存对象/结构体。启动时渲染第一帧。
2. **每次读取一个按键（或一行）**，分发给处理函数变更状态。
3. **每次操作后重新渲染**完整画面——不追加，而是替换。
4. **循环直到退出。**

整个画面应能在一屏内显示。

### 5. 一条命令即可运行

在项目现有任务运行器（`package.json` 脚本、`Makefile`、`justfile`、`pyproject.toml`）中添加脚本。用户应运行 `pnpm run <原型名称>` 或等价的命令——永远不需要记住路径。

如果宿主项目没有任务运行器，只需将命令放在原型 README 顶部。

### 6. 交付

给用户运行命令。他们自行操作；有趣的时刻是当他们说"等等，这应该不可能"或"哈，我以为 X 会不一样"的时候——这些就是*想法层面*的 bug，而这正是全部意义所在。如果他们想要添加新操作，就添加。原型是可以进化的。

### 7. 捕获答案

当原型完成使命后，问题的答案是唯一值得保留的东西。如果用户在场，询问他们学到了什么。如果不在，在原型旁边留下一个 `NOTES.md`，以便答案可以在原型被删除前填入（或者由你填入，如果你观看了整个过程）。

## 反模式

- **不要添加测试。** 需要测试的原型已不再是原型。
- **不要连接到真实数据库。** 使用内存存储，除非问题专门关于持久化。
- **不要泛化。** 不要"如果我们将来想支持 X 怎么办"。原型回答一个问题。
- **不要将逻辑和 TUI 混在一起。** 如果 reducer/状态机引用了 `console.log`、prompt 或终端转义码，它就不再可移植。保持 TUI 作为纯净模块之上的薄壳。
- **不要将 TUI 外壳发布到生产环境。** 外壳针对终端手动操作进行了优化。背后的逻辑模块才是值得保留的部分。
