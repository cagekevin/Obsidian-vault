# 代码架构审查

<what-to-do>
对 `HEAD` 与指定对照点之间的 diff，从以下五个维度执行深度架构级审查：

### 1. 需求对齐 (Requirement Alignment)
改动是否忠实实现了 S1 产出的 PRD 需求？是否偏离了 S2 看板/Issue 上定义的目标？
- 首先查找 Spec 来源：commit 消息中的 Issue 引用 → `kanban.html` 看板任务 → `docs/` 或 `.scratch/` 中的 PRD/文档 → 用户提供的文件。若无则跳过此维度。
- 有 Spec：对照检查——需求是否全部覆盖？有没有做了需求没要求的东西（scope creep）？实现方式是否符合 PRD 的 Implementation Decisions？
- 无 Spec：跳过此维度，其他四个维度正常执行。

### 2. 架构契合度 (Architecture)
代码改动是否违背项目的既定架构分层和模块边界？是否引入了不应该出现在该层的职责？
- ADR 中已有决策是否被遵守？
- 新模块放置位置是否合理？是扩展了系统能力还是破坏了原有抽象？

### 3. 数据一致性 (Data Consistency)
数据流是否完整无损？状态变更是否可追溯？
- 数据写入/更新/删除有无遗漏关联方？
- 有无部分更新导致数据不一致的风险？
- 有无混用不同层的数据模型（如把 DB Entity 直接吐给前端）？

### 4. 单一职责 (Single Responsibility)
每个模块/函数/组件是否只做一件事？
- 是否存在"顺便做了另一件事"的代码？
- 函数圈复杂度是否过高？能否拆解？
- 改动是否散落在不该改的模块里（scope creep）？

### 5. 全局最优 (Global Optimality)
这个改动对系统整体是净收益还是净负债？
- 有没有为了快速交付而牺牲可维护性？
- 有没有更好的、更整体的解决方案被忽略了？
- 新引入的依赖/模式/抽象是否值得其复杂度成本？

---

**执行流程：**
1. **确定对照点** — commit SHA、branch、tag 或 `main`。未指定则问用户。
2. **获取 diff** — `git diff <fixed-point>...HEAD`（三点 diff，对比 merge-base）。
3. **收集上下文** — 阅读 diff 涉及的模块代码、ADR、CONTEXT.md。查找 PRD/Spec 来源：commit 消息中的 Issue 引用 → `kanban.html` 看板 → `docs/`/`.scratch/` 中的文档 → 用户提供。
4. **运行评估** — 上述五维度逐维输出。不合并、不排序，让用户自己看各维度的权衡。
5. **输出报告**：
   - 每一维度一个区块，列出发现（含行号/代码段引用）。
   - 严重等级：🔴 架构违规 / 🟡 值得商榷 / 💡 建议优化。
   - 每个发现附带具体修改建议。

> 需求对齐维度视 Spec 可用性自动跳过：有 PRD/看板就对照查，没有就先过，不影响其他四个维度的审查。
</what-to-do>
