# 代码审查评估

在用户提供的固定点和 `HEAD` 之间的 diff 上进行双维度审查：

- **规范** —— 代码是否符合此仓库记录的编码规范？
- **规格** —— 代码是否忠实实现了原始 Issue / PRD / 规格？

两个维度作为**并行子 Agent**运行，这样它们不会污染彼此的上下文，然后此技能汇总它们的发现。

Issue 跟踪器应该已经提供给你了——如果 `docs/agents/issue-tracker.md` 缺失，运行 `/项目初始化设置`。

## 流程

### 1. 确定固定点

用户说的任何固定点——提交 SHA、分支名、标签、`main`、`HEAD~5` 等。不要自作主张；直接使用。如果用户没有指定，询问："从哪个点开始审查——分支、提交、还是 `main`？"直到得到答案才继续。

捕获 diff 命令：`git diff <fixed-point>...HEAD`（三点，所以比较的是合并基）。同时通过 `git log <fixed-point>..HEAD --oneline` 记录提交列表。

### 2. 确定规格来源

按此顺序查找原始规格：

1. 提交消息中的 Issue 引用（`#123`、`Closes #45`、GitLab `!67` 等）——通过 `docs/agents/issue-tracker.md` 中的工作流获取。
2. 用户作为参数传递的路径。
3. `docs/`、`specs/` 或 `.scratch/` 下与分支名或功能匹配的 PRD/规格文件。
4. 如果什么都没找到，询问用户规格在哪里。如果他们说没有，**规格**子 Agent 将跳过并报告"无可用规格"。

### 3. 确定规范来源

仓库中任何记录了代码应如何编写的资料。常见位置：

- `CLAUDE.md`、`AGENTS.md`
- `CONTRIBUTING.md`
- `CONTEXT.md`、`CONTEXT-MAP.md`、各上下文的 `CONTEXT.md` 文件
- `docs/adr/`（架构决策也是规范）
- `.editorconfig`、`eslint.config.*`、`biome.json`、`prettier.config.*`、`tsconfig.json`（工具强制执行的规范——注意它们，但不要重新检查工具已经检查过的内容）
- 仓库根目录或 `docs/` 下的任何 `STYLE.md`、`STANDARDS.md`、`STYLEGUIDE.md` 或类似文件

收集文件列表。**规范**子 Agent 将读取它们。

### 4. 并行生成两个子 Agent

发送一条消息，包含两个 `Agent` 工具调用。两个都使用 `general-purpose` 子 Agent。

**规范子 Agent 提示** —— 包括：

- 完整的 diff 命令和提交列表。
- 你在第 3 步找到的规范来源文件列表。
- 简要说明："阅读规范文档。然后阅读 diff。报告——按文件/hunk 报告——diff 中每个违反已记录规范的地方。引用规范（文件 + 规则）。区分硬违规和判断性意见。跳过工具已强制检查的内容。不超过 400 字。"

**规格子 Agent 提示** —— 包括：

- diff 命令和提交列表。
- 规格的路径或获取的内容。
- 简要说明："阅读规格。然后阅读 diff。报告：(a) 规格要求但缺失或不完整的功能；(b) diff 中未被要求的行为（范围蔓延）；(c) 看起来已实现但实现可能有问题的功能。对每个发现引用规格原文。不超过 400 字。"

如果规格缺失，跳过规格子 Agent 并在最终报告中说明。

### 5. 汇总

在 `## 规范` 和 `## 规格` 标题下呈现两个报告，逐字或轻微整理。**不要**合并或重新排序发现——两个维度故意分开，让用户可以独立查看。

以一行摘要结束：每个维度的发现总数，以及标记出的最严重问题（如果有的话）。

## 为什么需要两个维度

一个变更可能通过一个维度而失败于另一个：

- 遵循了每个规范但实现了错误功能的代码 → **规范通过，规格失败。**
- 完全按 Issue 要求做了但破坏了项目约定的代码 → **规格通过，规范失败。**

分开报告可以防止一个维度掩盖另一个维度。
