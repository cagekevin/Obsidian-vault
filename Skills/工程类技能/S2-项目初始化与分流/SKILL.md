# 项目基建与分流管家

<what-to-do>
作为通才，根据用户输入自动判定进入阶段 A（开荒初始化）或阶段 B（日常运维看板分流）。每一轮操作严格按本文件流程执行。

---

## 阶段 A：新项目开荒（项目初始化）
当用户要求"初始化项目"或首次调用时触发。**以下流程一次问一个问题，获得用户回答后再继续**，不要一次性全抛。

### 1. 环境探查
查阅当前仓库的起始状态，读取已有文件但不要假设内容：
- `git remote -v` 和 `.git/config` — 是否是 GitHub/GitLab 仓库
- `CLAUDE.md` / `AGENTS.md` 是否存在，是否已有 `## Agent skills` 段落
- `CONTEXT.md` / `CONTEXT-MAP.md` 是否存在
- `docs/adr/` 和 `docs/agents/` 是否已存在
- `.scratch/` 是否已在使用（本地 Markdown Issue Tracker 的标志）

### 2. 三个配置项（一次问一个）
每个配置项先给一句话解释（是什么、为什么需要），再展示选项和默认值。

**Section A — Issue Tracker**
> 解释：Issue Tracker 是 Issue 存放的地方。S1（需求拆分）、S2（本技能分流）、S6（TDD）都需要读写它。必须知道是用 `gh issue create` 调用 GitHub，还是写本地 .md 文件。
选项：GitHub（默认） / GitLab / 本地 Markdown（`.scratch/` 目录） / 其他（Jira/Linear 等——让用户用一段话描述工作流，以自由文本记录）

**Section B — Triage 标签词汇表**
> 解释：Triage 时 AI 需要在 Issue 上打标签。如果仓库已有不同的标签名（如 `bug:triage` 替代 `needs-triage`），在这里映射，避免重复创建。
五个标准角色：`needs-triage`、`needs-info`、`ready-for-agent`、`ready-for-human`、`wontfix`。默认=角色名等于标签名。问用户是否有覆盖。

**Section C — 领域文档布局**
> 解释：S6（TDD）、S9 等技能需要读取 CONTEXT.md 获取领域语言，以及 docs/adr/ 获取架构决策。需要知道是单上下文还是多上下文。
选项：单上下文（根目录一个 CONTEXT.md）/ 多上下文（根目录有 CONTEXT-MAP.md 指向子目录的 CONTEXT.md）

### 3. 预览与确认
向用户展示写入草稿：
- 要追加到 `CLAUDE.md` 或 `AGENTS.md` 的 `## Agent skills` 块
- 三个 `docs/agents/*.md` 文件内容
用户确认后再执行写入。

### 4. 资产落盘
**选择写入哪个文件：** 优先 CLAUDE.md（如果存在），否则 AGENTS.md。两者都存在时只编辑已有的那个，绝不重复创建。
将种子文件从本技能目录复制到目标仓库的 `docs/agents/`：
- `issue-tracker-github.md` / `issue-tracker-gitlab.md` / `issue-tracker-local.md` → 适配后写入 `issue-tracker.md`
- `triage-labels.md` → `triage-labels.md`
- `domain.md` → `domain.md`

### 5. 通电激活
在后台自动调用根目录下的 `skills-init.command` 脚本，一键生成项目的 `CLAUDE.md` 与 `.mdc` 技能加载器。完成后告知用户：设置已完成，`docs/agents/*.md` 可直接手动编辑，重新运行本技能仅用于切换 Issue Tracker 或重置。

## 阶段 B：日常运维（任务看板分流）
触发方式：用户输入 `/triage` 或要求"处理/盘点任务"。在所有外发注释顶部固定带上：`> *This was generated by AI during triage.*`

### 0. 模式检测
扫描项目根目录：
- **若存在 `kanban.html`**：进入 **Kanban 模式**。所有操作在 HTML DOM 上读写卡片（读取列、增删卡片、移动列间）。
- **若不存在**：进入 **Issue Tracker 模式**。走原有 GitHub/GitLab/Local Markdown 流程。
两种模式下，下文的状态机逻辑完全一致，仅操作载体不同。

### 前置约定
- **两类角色**：`bug`（有问题）/ `enhancement`（新功能或改进）
- **五种状态**：`needs-triage`、`needs-info`、`ready-for-agent`、`ready-for-human`、`wontfix`
- **状态机流向**：未打标签 → `needs-triage` → (`needs-info` / `ready-for-agent` / `ready-for-human` / `wontfix`)。`needs-info` 收到回复后回退到 `needs-triage`。维护者可在任何时刻覆盖。
- 每个 Issue 应恰好携带一个类别角色和一个状态角色。发生冲突时，提醒维护者。

### 看板总览
**Kanban 模式**：读取 `kanban.html`，解析出三列（待处理 / 进行中 / 已完成）中的所有卡片，按时间倒序展示给维护者。标记状态映射：待处理=`needs-triage`，进行中=`ready-for-agent/human`，已完成=`done`。
**Issue Tracker 模式**：查询 Issue Tracker，按时间倒序展示三个桶：未标记 / `needs-triage` / `needs-info` 且有回复。

无论哪种模式，显示计数和一行摘要，让维护者选择卡片/题号。

### 处理具体 Issue/Card
1. **收集上下文**：Issue 模式 — 阅读 Issue 正文/评论/标签。Kanban 模式 — 读取卡片的 title、desc、badge、blocked-by。查阅代码库，扫描 `.out-of-scope/*.md`。
2. **推荐**：给出分类和状态推荐（含理由），等待指示。
3. **复现（仅 Bug）**：在开始追问之前，优先尝试复现——追踪代码路径、运行测试或构建脚本。
4. **追问（如果需要）**：信息不足时转为 `needs-info`，模板见下文。提问必须具体可操作。
5. **执行最终状态转移**：
   - `ready-for-agent` — 追加 Agent Brief。**Kanban 模式**：在卡片 badge 标注 AFK，移入"进行中"列，追加验收标准到 desc。
   - `ready-for-human` — **Kanban 模式**：标注 HITL，移入"进行中"列，desc 注明无法自动派单的原因。
   - `needs-info` — 发布 Triage Notes。**Kanban 模式**：保持卡片在"待处理"列，追加 Triage Notes 到 desc。
   - `wontfix`（Bug）— 礼貌解释。**Kanban 模式**：移入"已完成"列，卡片标记灰色 `wontfix`。**Issue 模式**：直接关闭。
   - `wontfix`（Enhancement）— 写入 `.out-of-scope/`。**Kanban 模式**：同上移入"已完成"+wontfix 标记，并在 desc 添加 `.out-of-scope/` 链接。
   - `needs-triage` — 打标签。**Kanban 模式**：确认卡片在"待处理"列。

### 快速覆盖
**Issue 模式**：维护者说"把 #42 转 ready-for-agent"，直接信任执行，跳过追问。**Kanban 模式**：按卡片 ID 找到 DOM 元素，更新 badge 和列。

### 恢复上次会话
如果已有之前的 Triage Notes，读取它们，检查是否有新回复/变动后继续。不要重复问已解决的问题。

### Triage Notes 模板
```markdown
## Triage Notes
**已确定的事实：** [自动复现/阅读中发现的可信线索]
**需要补充的输入 (@reporter)：** [极其具体且具备可操作性的提问，拒绝模糊]
```
