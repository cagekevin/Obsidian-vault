# 快速原型构建

原型是**回答某个问题的一次性代码**。问题决定了形状。

## 选择一个分支

确定要回答哪个问题——从用户的提示、周围的代码，或如果用户在附近的话询问：

- **"这个逻辑/状态模型感觉对吗？"** → [LOGIC.md](LOGIC.md)。构建一个简单的交互式终端应用，推动状态机通过难以在纸上推理的案例。
- **"这应该长什么样？"** → [UI.md](UI.md)。在单一路由上生成几个截然不同的 UI 变体，通过 URL search param 和浮动底部栏切换。

两个分支产生非常不同的产物——选错了会浪费整个原型。如果问题确实模糊且用户联系不上，默认选择更适合周围代码的分支（后端模块 → 逻辑；页面或组件 → UI）并在原型顶部说明假设。

## 两者都适用的规则

1. **从第一天起就是一次性的，并明确标记。** 将原型代码放在实际使用位置附近（它正在为其做原型的模块或页面旁边），这样上下文显而易见——但命名要让随便一看的人都知道这是原型，不是生产代码。对于一次性 UI 路由，遵循项目已有的路由约定；不要发明新的顶层结构。
2. **一个命令即可运行。** 使用项目已有任务运行器支持的任何方式——`pnpm <name>`、`python <path>`、`bun <path>` 等。用户必须能无需思考就启动它。
3. **默认不持久化。** 状态存在内存中。持久化是原型要**检查**的东西，不是它应该依赖的东西。如果问题明确涉及数据库，连接一个临时数据库或带有清晰"PROTOTYPE — wipe me"名称的本地文件。
4. **跳过打磨。** 没有测试，没有超出使原型**可运行**的错误处理，没有抽象。重点是快速学到东西然后删掉它。
5. **暴露状态。** 每次动作（逻辑）或每次变体切换（UI）后，打印或渲染完整的相关状态，让用户看到什么发生了变化。
6. **完成后删除或吸收。** 当原型回答了它的问题时，要么删除它，要么将验证过的决策融入真实代码——不要让它烂在仓库里。

## 完成后

原型中唯一值得保留的是**答案**。将其捕获到某个持久的文档中（提交信息、ADR、Issue、或原型旁边的 `NOTES.md`），连同它所回答的问题。如果用户在附近，捕获就是一次快速对话；如果不在，留下占位符让他们（或你，下次做的时候）在删除原型前填入结论。
