# UI 原型

在单个路由上生成**若干在结构上截然不同的 UI 变体**，通过浮动的底部切换栏切换。用户在浏览器中翻转对比各变体，选择其中一个（或从每个变体中"偷"取一些元素），然后丢弃其余部分。

如果问题涉及的是逻辑/状态而非视觉呈现——走错了分支。请使用 [LOGIC.md](LOGIC.md)。

## 何时使用此模式

- "这个页面应该长什么样？"
- "我想在确定之前看看这个仪表盘的几种方案。"
- "为设置页面尝试不同的布局。"
- 任何当用户可能花一整天在心里纠结三种模糊线框图的场景。

## 两种子模式——强烈优先使用子模式 A

UI 原型在与**应用的其他部分衔接**时更容易评判——真实的头部、真实的侧边栏、真实的数据、真实的信息密度。孤立的临时路由如同真空——每个变体单独看都挺好。只要有可能将变体放在现有页面中，就默认使用子模式 A。只有当原型确实没有"近邻"页面可嵌入时，才选择子模式 B。

### 子模式 A——在现有页面上调整（推荐）

路由已存在。变体在**同一路由**上渲染，通过 `?variant=` URL 搜索参数控制。现有的数据获取、参数和认证逻辑保持不变——仅切换渲染部分。这是默认模式，除非有特定原因不这么做。

如果原型针对的是尚未有页面但**自然应存在于某个页面内**的内容（仪表盘的新区块、设置屏幕的新卡片、现有流程中的新步骤）——这仍然属于子模式 A。将变体挂载在宿主页面内。

### 子模式 B——新页面（最后手段）

仅当被原型化的内容确实没有现有页面可嵌入时使用——例如一个全新的顶层界面，或一个无法合理嵌入任何位置的流程。

创建一个**临时路由**，遵循项目已有的路由约定——不要发明新的顶层结构。命名要明显表明这是一个原型（例如在路径或文件名中包含 `prototype`）。同样使用 `?variant=` 模式。

在确定使用子模式 B 之前，进行合理性检查：真的没有现有页面可以嵌入吗？空路由会隐藏设计问题，而有内容的页面则会暴露它们。

两种子模式下，浮动底部切换栏是相同的。

## 流程

### 1. 明确问题并确定变体数 N

默认 **3 个变体**。超过 5 个就不再是"截然不同"而是噪音了——上限设置在此。

在原型位置或文件顶部注释中写下计划，一句话：

> "设置页面的三种变体，通过 `?variant=` 切换，在现有 `/settings` 路由上。"

无论用户是否反对，这个计划就已经明确。

### 2. 生成截然不同的变体

起草每个变体。每个变体需满足：

- 服务于页面的目的及其可访问的数据。
- 使用项目的组件库/样式系统（TailwindCSS、shadcn、MUI、纯 CSS 等）。
- 有清晰的导出组件名称，例如 `VariantA`、`VariantB`、`VariantC`。

变体必须在**结构上不同**——不同的布局、不同的信息层级、不同的主要交互方式，而不仅仅是颜色不同。三个略微调整的卡片网格不是 UI 原型，而是壁纸。如果两个起草结果过于相似，重新做一个，并明确要求"不要使用卡片网格"。

### 3. 将它们串联起来

在路由上创建一个切换器组件：

```tsx
// 伪代码——根据项目框架调整
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

对于子模式 A（现有页面）：保持所有现有数据获取逻辑在切换器之上；仅根据变体切换渲染的子组件树。

对于子模式 B（新页面）：临时路由位于 `/prototype/<name>` 下，挂载相同的切换器。

### 4. 构建浮动切换栏

一个固定在屏幕底部中央的小型栏，包含三个部分：

- **左箭头** — 切换到上一个变体（循环）。
- **变体标签** — 显示当前变体的键名，如果变体导出了名称，也显示该名称。例如 `B — 侧边栏布局`。
- **右箭头** — 切换到下一个变体（循环）。

行为：

- 点击箭头更新 URL 搜索参数（使用框架的路由器——Next.js 用 `router.replace`，React Router 用 `navigate` 等），使变体可分享且刷新后保持状态。
- 键盘：`←` 和 `→` 方向键也可切换。当 `<input>`、`<textarea>` 或 `[contenteditable]` 获得焦点时不拦截方向键。
- 在视觉上与页面明显区分（例如高对比度胶囊、微妙的阴影效果），以便明显看出它不是被评估的设计的一部分。
- 在生产构建中隐藏——使用 `process.env.NODE_ENV !== 'production'` 或等效检查来防止原型意外合并时将切换栏带到用户面前。

将切换器放入一个共享组件中，以便两种子模式均可复用。将其放置在项目中共享 UI 所在的位置。

### 5. 移交

提供 URL（以及 `?variant=` 键值）。用户随时可以翻阅查看。通常有趣的反馈是**"我想要 B 的头部加上 C 的侧边栏"**——那才是他们真正想要的设计。

### 6. 记录结论并清理

一旦某个变体胜出，记录是哪一个以及原因（提交信息、ADR、Issue，或者在 AFK 模式下用户尚未回应时，放在原型旁边的 `NOTES.md` 文件中）。然后：

- **子模式 A** — 删除落选的变体和切换器；将胜出变体合并到现有页面中。
- **子模式 B** — 将胜出变体提升为正式路由，删除临时路由和切换器。

不要留下变体组件或切换器。它们会迅速腐化并让后续读者感到困惑。

## 反模式

- **仅在颜色或文案上有差异的变体。** 那是微调，不是原型。真正的变体应该在结构上不同。
- **在变体之间共享过多代码。** 共享 `<Header>` 可以；共享 `<Layout>` 就失去了意义。每个变体应能自由替换布局。
- **将变体连接到真实的修改操作。** 只读原型就足够了。如果变体需要执行修改操作，指向一个桩（stub）——问题是"这应该长什么样"，而不是"后端能否工作"。
- **直接将原型提升到生产环境。** 变体代码是在原型约束下编写的（没有测试、最小化的错误处理）。在将其合并时应该重新正确编写。
