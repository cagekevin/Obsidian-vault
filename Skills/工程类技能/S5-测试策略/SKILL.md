# 测试策略

<what-to-do>

## 测试金字塔决策

| 层 | 速度 | 范围 | 信心来源 | 建议比例 |
|----|------|------|---------|---------|
| **单元测试** | 毫秒级 | 单个函数/模块 | 逻辑正确性 | ~70% |
| **集成测试** | 秒级 | 模块间交互 | 协作正确性 | ~20% |
| **E2E 测试** | 分钟级 | 完整用户流程 | 系统可用性 | ~10% |

## 何时用哪层

| 场景 | 推荐层 | 原因 |
|------|--------|------|
| 验证纯函数/算法 | 单元测试 | 快、确定、边界清晰 |
| 测试数据库交互 | 集成测试 | 需要真实 DB 行为 |
| 验证 API 端点 | 集成测试 | 覆盖路由+中间件+响应 |
| 用户登流程 | E2E 测试 | 涉及多系统协作 |
| Bug 回归验证 | 单元测试优先 | 最精确的回归信号 |

## Mock 边界

- **对外部系统**（DB、API、文件系统）— 在 seam 处 Mock
- **对内部模块** — 不要 Mock，用真实实现
- **Mock 太多 = 测试耦合实现细节** → 重构信号

## 覆盖率目标

- 核心业务逻辑：≥ 90%
- 基础设施代码：≥ 60%
- UI 组件：≥ 50%（重点关注交互逻辑）
